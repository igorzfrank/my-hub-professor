1 🧠 O que é Git e por que usar?
🎯 O que é Git?
Git é um sistema de controle de versão distribuído, gratuito e de código aberto. Ele registra o histórico de todas as alterações feitas nos arquivos do seu projeto, permitindo voltar a versões anteriores, trabalhar em equipe sem conflitos e manter um backup completo do código.
💡 Analogia simples
Git é como um botão "Salvar" turbinado de um videogame. Em vez de salvar por cima do arquivo anterior, você cria pontos de restauração (commits) com nome e descrição. Se algo der errado, você pode voltar a qualquer ponto anterior — como carregar um save antigo.
⭐Por que usar Git?
⏪ Histórico completo
Cada alteração fica registrada. Você pode voltar a qualquer versão anterior do projeto.
👥 Trabalho em equipe
Múltiplas pessoas editam o mesmo projeto simultaneamente sem conflitos.
🛡️ Segurança
Backup distribuído — cada pessoa tem o histórico completo do projeto.
🌳 Ramificações (branches)
Crie versões paralelas para testar novas funcionalidades sem quebrar o projeto principal.
🔍 Git ≠ GitHub: Git é o sistema de controle de versão que roda no seu computador. GitHub é uma plataforma online que hospeda repositórios Git na nuvem para compartilhamento e colaboração.
2 ⚙️ Configuração Inicial do Git
Antes de começar a usar o Git, você precisa configurar seu nome e e-mail. Essas informações aparecerão em todos os seus commits.
📝Comandos de configuração
# Configurar seu nome
git config --global user.name "Seu Nome Completo"

# Configurar seu e-mail
git config --global user.email "seuemail@exemplo.com"

# Verificar as configurações
git config --list
💡 Dica: O parâmetro --global aplica a configuração para todos os projetos no seu computador. Sem ele, a configuração vale apenas para o projeto atual.
3 🔄 O Fluxo Básico do Git — Os 4 Estados
Entender o fluxo de trabalho do Git é essencial. Seu arquivo passa por 4 estados diferentes até ser salvo no histórico:
📂
Working Directory
Arquivos editados
→
📦
Staging Area
git add
→
💾
Repositório Local
git commit
→
☁️
Repositório Remoto
git push
1. Working Directory (Diretório de Trabalho)
Seus arquivos normais, onde você edita o código. O Git ainda não está rastreando as mudanças.
2. Staging Area (Área de Preparação)
Você seleciona quais arquivos farão parte do próximo commit. Use git add.
3. Repositório Local
O commit salva as mudanças no histórico do Git no seu computador. Use git commit.
4. Repositório Remoto
Envia os commits para a nuvem (GitHub, GitLab). Use git push.
4 📋 Comandos Básicos do Git — Guia Prático
1git init — Inicializar um repositório
Cria um novo repositório Git na pasta atual. Uma pasta oculta .git será criada com todo o controle de versão.
git init
2git status — Verificar o estado dos arquivos
Mostra quais arquivos foram modificados, quais estão na área de preparação e quais não estão sendo rastreados. Use com frequência!
git status
3git add — Adicionar arquivos à área de preparação
Move os arquivos do diretório de trabalho para a Staging Area. Você escolhe o que fará parte do próximo commit.
# Adicionar um arquivo específico
git add index.html

# Adicionar todos os arquivos da pasta
git add .
4git commit — Salvar no histórico
Cria um ponto de restauração (commit) com uma mensagem descritiva. Este é o coração do Git.
# Commit com mensagem curta
git commit -m "Adiciona cabeçalho e rodapé"
5git log — Ver o histórico de commits
Exibe a lista de todos os commits do repositório em ordem cronológica reversa (mais recente primeiro).
git log
# Versão compacta (uma linha por commit)
git log --oneline
6git remote add + git push — Conectar e enviar para o GitHub
Vincula seu repositório local a um repositório remoto e envia os commits para a nuvem.
# Conectar ao repositório remoto (execute apenas 1 vez)
git remote add origin "url-do-repositorio"

# Enviar commits para o GitHub
git push -u origin main
7git clone — Copiar um repositório existente
Baixa uma cópia completa de um repositório remoto para o seu computador. Útil para começar a trabalhar em projetos existentes.
git clone "url-do-repositorio"
8git pull — Atualizar seu repositório local
Baixa e integra as alterações do repositório remoto para o seu repositório local. Use sempre antes de começar a trabalhar.
git pull origin main
5 🔄 Fluxo de Trabalho Completo — Do Início ao Push
Este é o ciclo que você repetirá todas as vezes que for trabalhar com Git. Memorize esta sequência:
# 1. Crie ou entre na pasta do projeto
cd meu-projeto

# 2. Inicialize o Git (apenas 1 vez)
git init

# 3. Verifique o status (use sempre!)
git status

# 4. Adicione os arquivos à área de preparação
git add .

# 5. Faça o commit com uma mensagem clara
git commit -m "Meu primeiro commit"

# 6. Conecte ao GitHub (apenas 1 vez)
git remote add origin "url-do-repositorio"

# 7. Envie para a nuvem
git push -u origin main

# 8. Nas próximas vezes, apenas repita os passos 3, 4, 5 e 7
💡 Regra de ouro: Sempre faça git status antes de qualquer comando. Ele mostra exatamente o que está acontecendo e evita surpresas. Também é uma boa prática fazer git pull antes de começar a trabalhar para garantir que está na versão mais recente.
6 ⚠️ Erros Comuns e Boas Práticas
❌ Erros comuns
❌ Esquecer de fazer git add antes do commit
❌ Mensagens de commit vagas: "correções", "ajustes"
❌ Fazer commit de arquivos grandes (imagens, vídeos)
❌ Esquecer de dar git pull antes de começar
❌ Fazer push de credenciais/senhas para o GitHub
❌ Criar repositório Git dentro de outro repositório Git
✅ Boas práticas
✅ Commits pequenos e frequentes (cada alteração lógica)
✅ Mensagens descritivas no imperativo: "Adiciona formulário de login"
✅ Use .gitignore para excluir arquivos desnecessários
✅ Sempre faça git status antes de qualquer comando
✅ Mantenha um README.md explicando o projeto
✅ Faça backup remoto regularmente (push)
7 💡 Dicas Extras e Atalhos
📄 Arquivo .gitignore — O que NÃO versionar
Nem tudo deve ir para o Git. Crie um arquivo .gitignore para excluir pastas e arquivos desnecessários:
# Exemplo de .gitignore
node_modules/
.env
*.log
arquivos-temporarios/
