Módulo 1
O problema dos dados em memória
1.1 — Por que o array não é suficiente?
Nas aulas anteriores, o Model armazenava os dados em um array dentro do próprio código. Isso é ótimo para aprender, mas tem um problema fatal: quando o servidor é reiniciado, tudo que foi criado, editado ou excluído desaparece. Uma API real precisa que os dados sobrevivam — é aí que entra o banco de dados.
Antes — Array em memória
// produto.model.js
let produtos = [
  { id: 1, nome: 'Teclado' }
];

export default produtos;
  • Dados somem ao reiniciar
  • Sem pesquisa eficiente
  • Sem concorrência de usuários
Depois — Banco de dados
-- tabela no PostgreSQL
CREATE TABLE produtos (
  id   SERIAL PRIMARY KEY,
  nome TEXT NOT NULL
);
  • Dados persistem para sempre
  • Consultas poderosas com SQL
  • Suporta muitos usuários
1.2 — O que é o PostgreSQL e o pacote pg?
PostgreSQL é um banco de dados relacional — armazena dados em tabelas com linhas e colunas, e usa SQL para consultas. Para o Node.js se comunicar com ele, usamos o pacote pg (chamado de node-postgres), que é o driver oficial.
Analogia: se o PostgreSQL é um banco — um lugar físico que guarda dinheiro — o pacote pg é o caixa eletrônico que dá acesso a esse banco. Você não entra no banco diretamente; você usa a interface que o caixa (driver) oferece.
O pg oferece duas formas de conectar: Client (uma conexão única) e Pool (um conjunto de conexões reutilizáveis). Em APIs web, sempre usamos o Pool — ele gerencia automaticamente múltiplas requisições simultâneas sem abrir uma nova conexão para cada uma.

2.2 — O arquivo de conexão: db.js
Este é o arquivo mais importante desta aula. Ele cria um Pool de conexões com o PostgreSQL e o exporta para ser usado em todos os Models. Só precisamos configurar isso uma vez — todos os outros arquivos simplesmente importam o pool pronto.
src/database/db.js
import pg from 'pg';
const { Pool } = pg;

// Configuração da conexão com o banco
const pool = new Pool({
  host: 'localhost', // endereço do servidor do banco
  port: 5432, // porta padrão do PostgreSQL
  database: 'minha_api', // nome do banco de dados
  user: 'postgres', // usuário do banco
  password: 'senha123', // senha do usuário
});

// Exporta o pool para uso nos Models
export default pool;
O que é um Pool de conexões?
Abrir uma conexão com o banco a cada requisição seria lento e caro. O Pool mantém um conjunto de conexões abertas e reutilizáveis. Quando uma requisição chega, o Pool empresta uma conexão livre; quando termina, devolve ao conjunto. Por padrão, o pg cria um Pool com até 10 conexões simultâneas.
2.3 — Criando a tabela no PostgreSQL
Antes de escrever qualquer código Node.js, precisamos criar o banco de dados e a tabela no PostgreSQL. Execute estes comandos no pgAdmin ou no terminal do psql:
SQL — executar no pgAdmin ou psql
-- 1. Cria o banco de dados
CREATE DATABASE minha_api;

-- 2. Conecta ao banco (no psql: \c minha_api)

-- 3. Cria a tabela de produtos
CREATE TABLE produtos (
  id         SERIAL PRIMARY KEY, -- ID automático
  nome       TEXT NOT NULL, -- nome obrigatório
  preco      NUMERIC(10, 2) NOT NULL, -- preço com 2 decimais
  criado_em  TIMESTAMP DEFAULT NOW() -- data automática
);

-- 4. Insere dados de exemplo

Módulo 3
O Model com banco de dados — async/await
3.1 — Por que usamos async/await?
Consultas ao banco de dados não são instantâneas — o Node.js precisa enviar a query pela rede, esperar o PostgreSQL processar e receber a resposta. Esse tempo de espera é chamado de operação assíncrona. Se não tratarmos isso corretamente, o código tentará usar o resultado antes de ele ter chegado.
Sem async/await — ERRADO
function buscarTodos() {
  const result = pool.query('SELECT * FROM produtos');
  return result.rows; // ← undefined!
  // result ainda não chegou
}
Com async/await — CORRETO
async function buscarTodos() {
  const result = await pool.query('SELECT * FROM produtos');
  return result.rows; // ← dados reais
  // espera a resposta chegar
}
Regra prática: toda função que faz pool.query() deve ser async. E toda chamada a essa função no Controller deve usar await. Sempre em par.
3.2 — Como a query funciona — result.rows
O método pool.query() retorna um objeto com várias propriedades. As mais importantes para nós são:
Propriedade O que contém Quando usar
result.rows Array com os registros retornados SELECT — buscar dados
result.rows[0] O primeiro (e único) registro INSERT/SELECT por ID
result.rowCount Número de linhas afetadas DELETE — checar se existia
3.3 — O Model com banco de dados
O Model agora não tem mais o array. Em vez disso, exporta funções assíncronas que executam queries SQL. Cada operação do CRUD vira uma função própria.
src/models/produto.model.js
import pool from '../database/db.js';

// ── READ (todos) ────────────────────────────────────
export async function buscarTodos() {
  const result = await pool.query('SELECT * FROM produtos ORDER BY id');
  return result.rows;
}

// ── READ (um) ────────────────────────────────────────
export async function buscarPorId(id) {
  const result = await pool.query(
    'SELECT * FROM produtos WHERE id = $1', [id]
  );
  return result.rows[0]; // retorna o objeto ou undefined
}

// ── CREATE ───────────────────────────────────────────
export async function criar(nome, preco) {
  const result = await pool.query(
    'INSERT INTO produtos (nome, preco) VALUES ($1, $2) RETURNING *',
    [nome, preco]
  );
  return result.rows[0]; // retorna o produto criado (com o id gerado)
}

// ── UPDATE ───────────────────────────────────────────
export async function atualizar(id, nome, preco) {
  const result = await pool.query(
    'UPDATE produtos SET nome = $1, preco = $2 WHERE id = $3 RETURNING *',
    [nome, preco, id]
  );
  return result.rows[0];
}

// ── DELETE ───────────────────────────────────────────
export async function remover(id) {
  const result = await pool.query(
    'DELETE FROM produtos WHERE id = $1', [id]
  );
  return result.rowCount; // 1 se deletou, 0 se não existia
}
Por que $1, $2 em vez de colar o valor direto?
Os $1, $2 são parâmetros de query. Nunca coloque valores do usuário diretamente dentro da string SQL — isso abre uma vulnerabilidade chamada SQL Injection. Os parâmetros fazem o pg tratar os valores como dados seguros, nunca como código SQL.
Módulo 4
O Controller com banco de dados — try/catch
4.1 — Por que precisamos do try/catch?
Quando usamos await, se algo der errado (banco fora do ar, query errada, violação de constraint) o Node.js lança um erro. Sem o try/catch, esse erro derrubaria o servidor inteiro. O try/catch nos permite capturar o erro e retornar um status 500 adequado ao cliente.
src/controllers/produto.controller.js
import * as ProdutoModel from '../models/produto.model.js';

// ── Listar todos ─────────────────────────────────────
export async function listar(req, res) {
  try {
    const produtos = await ProdutoModel.buscarTodos();
    res.json(produtos);
  } catch (erro) {
    res.status(500).json({ erro: 'Erro ao buscar produtos' });
  }
}

// ── Buscar por ID ────────────────────────────────────
export async function buscarPorId(req, res) {
  try {
    const produto = await ProdutoModel.buscarPorId(req.params.id);
    if (!produto) {
      return res.status(404).json({ erro: 'Produto não encontrado' });
    }
    res.json(produto);
  } catch (erro) {
    res.status(500).json({ erro: 'Erro ao buscar produto' });
  }
}

// ── Criar ────────────────────────────────────────────
export async function criar(req, res) {
  try {
    const { nome, preco } = req.body;
    if (!nome || !preco) {
      return res.status(400).json({ erro: 'nome e preco são obrigatórios' });
    }
    const novoProduto = await ProdutoModel.criar(nome, preco);
    res.status(201).json(novoProduto);
  } catch (erro) {
    res.status(500).json({ erro: 'Erro ao criar produto' });
  }
}

// ── Atualizar ────────────────────────────────────────
export async function atualizar(req, res) {
  try {
    const { nome, preco } = req.body;
    const produto = await ProdutoModel.atualizar(req.params.id, nome, preco);
    if (!produto) {
      return res.status(404).json({ erro: 'Produto não encontrado' });
    }
    res.json(produto);
  } catch (erro) {
    res.status(500).json({ erro: 'Erro ao atualizar produto' });
  }
}

// ── Remover ──────────────────────────────────────────
export async function remover(req, res) {
  try {
    const deletados = await ProdutoModel.remover(req.params.id);
    if (deletados === 0) {
      return res.status(404).json({ erro: 'Produto não encontrado' });
    }
    res.status(204).send();
  } catch (erro) {
    res.status(500).json({ erro: 'Erro ao remover produto' });
  }
}

Módulo 5
Routes e app.js — sem mudanças!
Essa é a grande vantagem do MVC: o arquivo de rotas e o app.js não precisam saber se os dados vêm de um array ou de um banco de dados. Eles só conhecem o Controller — e o Controller não mudou de interface, só de implementação interna.
src/routes/produto.routes.js (igual ao anterior)
import { Router } from 'express';
import * as C from '../controllers/produto.controller.js';

const router = Router();

router.get ('/', C.listar);
router.get ('/:id', C.buscarPorId);
router.post ('/', C.criar);
router.put ('/:id', C.atualizar);
router.delete('/:id', C.remover);

export default router;
app.js (igual ao anterior)
import express from 'express';
import produtoRoutes from './src/routes/produto.routes.js';

const app = express();
app.use(express.json());

app.use('/produtos', produtoRoutes);

app.listen(3000, () => {
  console.log('Servidor rodando!');
});
Módulo 6
Conceitos essenciais de SQL com pg
6.1 — RETURNING * — devolvendo o registro criado
Por padrão, o INSERT e o UPDATE do PostgreSQL não retornam os dados do registro alterado — apenas informam quantas linhas foram afetadas. Adicionando RETURNING * no final da query, o banco devolve o registro completo (incluindo o ID gerado automaticamente).
Sem RETURNING — não sabemos o ID gerado
INSERT INTO produtos (nome, preco)
VALUES ('Teclado', 350);

-- result.rows = []
-- result.rowCount = 1
Com RETURNING * — recebemos o registro completo
INSERT INTO produtos (nome, preco)
VALUES ('Teclado', 350)
RETURNING *;

-- result.rows[0] = { id: 4, nome: 'Teclado', preco: 350 }
6.2 — Queries mais comuns no dia a dia
Operação Query SQL O que usar do result
Buscar todos SELECT * FROM tabela result.rows
Buscar um SELECT * FROM tabela WHERE id = $1 result.rows[0]
Inserir INSERT INTO tabela (col) VALUES ($1) RETURNING * result.rows[0]
Atualizar UPDATE tabela SET col = $1 WHERE id = $2 RETURNING * result.rows[0]
Deletar DELETE FROM tabela WHERE id = $1 result.rowCount
