Módulo 1
O problema de colocar senhas no código
1.1 — O que fizemos até agora e por que é arriscado
Na aula anterior, o arquivo db.js ficou assim — com a senha do banco escrita diretamente no código:
src/database/db.js — PROBLEMA: dados sensíveis expostos no código
import pg from 'pg';
const { Pool } = pg;

const pool = new Pool({
  host: 'localhost', // ← exposto
  port: 5432,
  database: 'minha_api', // ← exposto
  user: 'postgres', // ← exposto
  password: 'senha123', // ← SENHA EXPOSTA NO CÓDIGO!
});

export default pool;
1.2 — Por que isso é um problema sério
Quando o projeto está em um repositório Git — como no GitHub — todos que tiverem acesso ao repositório verão a senha do banco de dados. Mesmo que o repositório seja privado hoje, qualquer descuido pode expô-lo. No mercado de trabalho, nunca se coloca credencial no código.
Riscos de senha no código
  • Qualquer pessoa com acesso ao Git vê a senha
  • A senha fica registrada no histórico de commits
  • Se o repositório vazar, o banco fica exposto
  • Difícil de trocar a senha sem alterar o código
Com variáveis de ambiente
  • Credenciais ficam fora do código-fonte
  • O arquivo .env nunca vai para o Git
  • Cada ambiente (dev, produção) tem suas próprias configs
  • Trocar a senha não exige alterar nenhum arquivo de código
Analogia: colocar a senha do banco no código é como escrever a senha do Wi-Fi na parede da sala — qualquer visitante que entrar na casa (repositório) vai ver. O arquivo .env é como um cofre guardado em outro lugar: só quem está na máquina tem acesso.
Módulo 2
O arquivo .env e o pacote dotenv
2.1 — O que é uma variável de ambiente?
Uma variável de ambiente é uma informação armazenada no sistema operacional ou em um arquivo especial, fora do código. O Node.js permite acessar essas variáveis pelo objeto global process.env. Qualquer variável definida no ambiente aparece como uma propriedade desse objeto.
Acessando variáveis de ambiente no Node.js
// process.env contém todas as variáveis de ambiente do sistema
console.log(process.env.DB_HOST); // 'localhost'
console.log(process.env.DB_PASSWORD); // 'senha123'
console.log(process.env.PORT); // '3000'
2.2 — O que é o arquivo .env?
O arquivo .env é um arquivo de texto simples criado na raiz do projeto. Cada linha define uma variável no formato NOME=valor, sem espaços ao redor do = e sem aspas (em geral).
.env — na raiz do projeto
# Configurações do servidor
PORT=3000

# Configurações do banco de dados
DB_HOST=localhost
DB_PORT=5432
DB_NAME=minha_api
DB_USER=postgres
DB_PASSWORD=senha123
2.3 — O que é o pacote dotenv?
Por padrão, o Node.js não lê o arquivo .env automaticamente. O pacote dotenv faz exatamente isso: ele lê o arquivo .env e injeta cada variável dentro de process.env, tornando-as disponíveis para todo o código.
Terminal — instalação
npm install dotenv
Atenção — ES Modules e dotenv
Com "type": "module" no package.json, usamos a sintaxe de importação especial do dotenv: import 'dotenv/config'. Essa linha carrega o .env automaticamente — sem precisar chamar nenhuma função depois. Ela deve ser a primeira linha do arquivo principal.
Módulo 3
Como usar o .env no projeto — passo a passo
3.1 — A nova estrutura de arquivos
Adicionamos dois arquivos novos na raiz do projeto: o .env (que fica na máquina, fora do Git) e o .env.example (que vai para o Git como modelo vazio para outros desenvolvedores).
Estrutura do projeto
projeto/
 ├── src/
 │   ├── database/ → db.js
 │   ├── models/
 │   ├── controllers/
 │   └── routes/
 ├── app.js
 ├── .env          ← NÃO vai para o Git (contém as senhas)
 ├── .env.example  ← VAI para o Git (modelo sem valores)
 └── .gitignore    ← garante que .env seja ignorado
3.2 — O arquivo .gitignore
O .gitignore diz ao Git quais arquivos ele deve ignorar. Sempre adicione .env nele — sem isso, suas credenciais seriam enviadas ao repositório.
.gitignore — na raiz do projeto
# dependências instaladas
node_modules/

# variáveis de ambiente — NUNCA enviar ao Git
.env
3.3 — O arquivo .env.example
O .env.example é um modelo do .env sem os valores reais. Ele vai para o Git e serve como documentação: qualquer pessoa que clonar o projeto saberá quais variáveis precisam ser configuradas.
.env.example — vai para o Git, sem valores reais
# Configurações do servidor
PORT=

# Configurações do banco de dados
DB_HOST=
DB_PORT=
DB_NAME=
DB_USER=
DB_PASSWORD=
3.4 — O app.js — importando o dotenv primeiro
O import 'dotenv/config' deve ser a primeira linha do arquivo app.js. Isso garante que todas as variáveis do .env estejam disponíveis em process.env antes de qualquer outro arquivo ser importado — incluindo o db.js.
app.js
import 'dotenv/config';   // ← PRIMEIRA linha — carrega o .env
import express from 'express';
import produtoRoutes from './src/routes/produto.routes.js';

const app = express();
app.use(express.json());
app.use('/produtos', produtoRoutes);

// PORT vem do .env — com fallback para 3000 caso não esteja definida
const PORT = process.env.PORT || 3000;

app.listen(PORT, () => {
  console.log(`Servidor rodando na porta ${PORT}`);
});
3.5 — O db.js — usando process.env
Agora o db.js não tem mais nenhuma credencial. Cada valor vem de process.env, que já foi populado pelo dotenv quando o app.js iniciou.
src/database/db.js — CORRETO: sem nenhuma credencial no código
import pg from 'pg';
const { Pool } = pg;

const pool = new Pool({
  host: process.env.DB_HOST,
  port: process.env.DB_PORT,
  database: process.env.DB_NAME,
  user: process.env.DB_USER,
  password: process.env.DB_PASSWORD,
});

export default pool;
Módulo 4
Como tudo funciona junto — o fluxo completo
4.1 — O que acontece quando o servidor inicia
1
 
Node.js executa o app.js
A primeira linha import 'dotenv/config' é processada imediatamente.
2
 
dotenv lê o arquivo .env
Cada linha é lida e injetada em process.env. Agora process.env.DB_PASSWORD tem o valor que estava no arquivo.
3
 
Os demais imports são processados
O db.js é importado e lê process.env.DB_* para criar o Pool — os valores já estão disponíveis.
4
Servidor inicia com segurança
O Pool conecta ao banco usando as credenciais do .env. Nenhuma senha aparece em nenhum arquivo do código.
4.2 — Comparação: antes e depois
Aspecto Antes (senha no código) Depois (com .env)
Senha no repositório Sim — visível no db.js Não — fica só na máquina
Trocar a senha Editar o código e fazer commit Editar só o .env, sem tocar no código
Ambiente de produção Requer alterar o código para cada servidor Basta criar um .env diferente no servidor
Outro dev clona o projeto Recebe as credenciais junto com o código Copia o .env.example e preenche com as suas

Models, Controllers e Routes — sem alteração
Os arquivos *.model.js, *.controller.js e *.routes.js não precisam de nenhuma alteração. Eles apenas importam o pool do db.js, que já está configurado com as variáveis de ambiente — sem saber nem precisar saber de onde vêm os valores.
 
Pontos de atenção
import 'dotenv/config' deve ser a primeira linha do app.js
Se qualquer outro import aparecer antes, o db.js pode ser carregado antes de process.env ser populado — e o Pool será criado com valores undefined.
Valores do process.env são sempre strings
Mesmo que você escreva DB_PORT=5432 no .env, process.env.DB_PORT será a string '5432'. O pacote pg já converte a porta para número internamente, então para este caso não há problema.
Nunca adicione aspas nos valores do .env
Escreva DB_PASSWORD=senha123, não DB_PASSWORD="senha123". Com aspas, elas fazem parte do valor — e a senha que chegaria ao banco seria "senha123" com as aspas inclusas.
Reinicie o servidor ao alterar o .env
O dotenv lê o arquivo apenas uma vez, quando o servidor inicia. Se você alterar o .env com o servidor rodando, pare e inicie novamente para as mudanças surtirem efeito.
