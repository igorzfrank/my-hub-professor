Glossário de Jargões do Desenvolvimento de Sistemas
Terminologias essenciais utilizadas no dia a dia de equipes de desenvolvimento de software.
Capacidades Trabalhadas
• Selecionar ferramentas de gerenciamento na aplicação da metodologia
• Identificar metodologia de desenvolvimento de sistemas
Sobre Este Glossário

O desenvolvimento de software possui uma linguagem própria, com termos técnicos que facilitam a comunicação entre profissionais. Este glossário apresenta os principais jargões utilizados no dia a dia de equipes ágeis, organizados por categorias para facilitar o entendimento e a consulta.

🔄
Metodologias Ágeis
Sprint
Ciclo de desenvolvimento com duração fixa (geralmente 1-4 semanas) onde a equipe trabalha em um conjunto de tarefas definidas. Ao final, há uma entrega incremental do produto.
Backlog
Lista priorizada de todas as funcionalidades, melhorias e correções que precisam ser desenvolvidas no produto. Gerenciado pelo Product Owner.
Daily / Stand-up
Reunião diária rápida (15 min) onde cada membro compartilha: o que fez ontem, o que fará hoje e se há impedimentos. Promove alinhamento do time.
Retrospectiva (Retro)
Reunião ao final da sprint para refletir sobre o que funcionou bem, o que pode melhorar e definir ações de melhoria contínua para a próxima sprint.
Planning / Sprint Planning
Reunião no início da sprint onde a equipe define quais itens do backlog serão trabalhados e como serão executados.
User Story
Descrição simples de uma funcionalidade do ponto de vista do usuário. Formato: "Como [papel], quero [ação] para [benefício]".
Velocity
Métrica que mede a quantidade de trabalho (story points) que a equipe consegue completar em uma sprint. Usada para planejamento.
Definition of Done (DoD)
Checklist de critérios que uma tarefa deve cumprir para ser considerada completa (testes, code review, documentação, etc).
💻
Desenvolvimento e Código
Code Review
Processo onde outros desenvolvedores revisam o código antes de ser integrado, verificando qualidade, padrões e possíveis bugs.
Refatoração (Refactoring)
Reestruturação do código para melhorar sua legibilidade, performance ou manutenibilidade, sem alterar seu comportamento externo.
Pair Programming
Técnica onde dois desenvolvedores trabalham juntos no mesmo código: um escreve (driver) e outro revisa e sugere (navigator).
Hardcoded
Valores fixos inseridos diretamente no código-fonte, ao invés de usar variáveis ou configurações. Geralmente considerado má prática.
Clean Code
Código bem escrito, legível e de fácil manutenção. Segue boas práticas, convenções e é autoexplicativo.
Technical Debt (Débito Técnico)
Custo futuro de refatoração causado por soluções rápidas ou código mal escrito. Precisa ser "pago" eventualmente.
Boilerplate
Código repetitivo e padrão que precisa ser escrito em muitos lugares. Templates ou frameworks ajudam a reduzir boilerplate.
Merge Conflict
Conflito que ocorre quando dois desenvolvedores modificam as mesmas linhas de código e o Git não consegue unir automaticamente.
🔀
Controle de Versão (Git)
Commit
Snapshot do código em um momento específico. Registra mudanças no repositório com uma mensagem descritiva.
Branch
Ramificação do código principal onde desenvolvedores trabalham em funcionalidades isoladas sem afetar a branch principal (main/master).
Pull Request (PR) / Merge Request
Solicitação para integrar código de uma branch para outra. Permite revisão antes da integração.
Push
Enviar commits locais para o repositório remoto (GitHub, GitLab, Bitbucket).
Pull
Baixar e integrar mudanças do repositório remoto para o repositório local.
Rebase
Reescrever o histórico de commits movendo uma branch para uma nova base. Mantém histórico linear e limpo.
Cherry-pick
Aplicar um commit específico de uma branch em outra, sem fazer merge completo.
Stash
Salvar temporariamente mudanças não commitadas para limpar o working directory sem perder o trabalho.
🧪
Testes e Qualidade
Bug
Erro ou defeito no software que causa comportamento inesperado ou incorreto. Precisa ser corrigido.
Hotfix
Correção urgente de bug crítico em produção. Bypass do processo normal para resolver problema imediato.
Mock
Objeto simulado usado em testes para substituir dependências reais (APIs, banco de dados) e isolar o código testado.
Unit Test (Teste Unitário)
Teste de pequenas unidades de código (funções, métodos) de forma isolada para garantir que funcionam corretamente.
Integration Test (Teste de Integração)
Teste que verifica se diferentes módulos ou serviços funcionam corretamente quando integrados.
E2E Test (End-to-End)
Teste que simula o fluxo completo do usuário, desde a interface até o banco de dados, validando todo o sistema.
Smoke Test
Teste superficial e rápido para verificar se as funcionalidades principais estão funcionando após um deploy.
Regression Test (Teste de Regressão)
Teste para garantir que novas alterações não quebraram funcionalidades existentes.
🚀
Ambientes e Deploy
Deploy
Processo de publicar uma nova versão do software em um ambiente (desenvolvimento, homologação ou produção).
Staging / Homologação
Ambiente que replica produção onde testes finais são realizados antes do deploy em produção. Validação do cliente/PO acontece aqui.
Produção (Production / Prod)
Ambiente real onde usuários finais acessam o sistema. Qualquer erro aqui impacta diretamente os usuários.
Desenvolvimento (Dev)
Ambiente local ou compartilhado onde desenvolvedores testam suas alterações antes de enviar para staging.
Sandbox
Ambiente isolado para experimentação e testes sem risco de afetar outros ambientes. Útil para testes de APIs externas.
Rollback
Reverter o sistema para uma versão anterior estável quando um deploy causa problemas em produção.
CI/CD (Continuous Integration/Deployment)
Prática de integrar código frequentemente e automatizar testes e deploys. Acelera entregas e reduz erros.
Blue-Green Deployment
Estratégia de deploy onde duas versões idênticas do ambiente existem. Uma está ativa (blue) e a outra recebe a atualização (green). Troca instantânea minimiza downtime.
☁️
Infraestrutura e DevOps
Container (Docker)
Pacote leve e portátil que contém aplicação e todas suas dependências. Garante que o software roda igual em qualquer ambiente.
Kubernetes (K8s)
Plataforma de orquestração de containers. Gerencia deploy, escala e operação de containers automaticamente.
On-premise (On-prem)
Infraestrutura física localizada nas instalações da empresa (servidores próprios), ao invés de usar cloud.
Cloud
Serviços de computação, armazenamento e infraestrutura hospedados remotamente (AWS, Azure, GCP). Escalável e pago por uso.
Serverless
Modelo onde o desenvolvedor não gerencia servidores. Código executa em resposta a eventos e é cobrado apenas pelo uso.
Monitoramento (Monitoring)
Acompanhamento contínuo da saúde e performance do sistema (uso de CPU, memória, erros). Ferramentas: Grafana, Datadog, New Relic.
Load Balancer
Distribui requisições entre múltiplos servidores para equilibrar carga e evitar sobrecarga em um único servidor.
Scalability (Escalabilidade)
Capacidade do sistema de lidar com aumento de carga. Horizontal: adicionar servidores. Vertical: aumentar recursos do servidor.
🏗️
Arquitetura e Stack
Stack (Tech Stack)
Conjunto de tecnologias, linguagens, frameworks e ferramentas usadas no projeto. Exemplo: MERN (MongoDB, Express, React, Node).
Full Stack
Desenvolvedor que trabalha tanto no front-end quanto no back-end, tendo conhecimento de toda a stack.
API (Application Programming Interface)
Interface que permite comunicação entre sistemas. Define como solicitar e receber dados de um serviço.
REST / RESTful API
Estilo arquitetural para APIs que usa HTTP e operações padrão (GET, POST, PUT, DELETE) para manipular recursos.
GraphQL
Linguagem de consulta para APIs. Cliente especifica exatamente quais dados precisa, evitando over-fetching.
Microservices (Microsserviços)
Arquitetura onde aplicação é dividida em pequenos serviços independentes que se comunicam via APIs. Oposto de monolito.
Monolito (Monolithic)
Arquitetura onde toda a aplicação é um único sistema acoplado. Simples de desenvolver, mas difícil de escalar.
Middleware
Camada de software entre aplicação e sistema operacional ou entre cliente e servidor. Processa requisições (autenticação, logging).
⚡
Performance e Otimização
Cache
Armazenamento temporário de dados frequentemente acessados para acelerar respostas. Reduz chamadas ao banco de dados.
Lazy Loading
Técnica de carregar recursos (imagens, componentes) apenas quando necessários, melhorando tempo de carregamento inicial.
Throttling
Limitar a frequência de execução de uma função. Útil em eventos que disparam muitas vezes (scroll, resize).
Debouncing
Atrasar a execução de função até que eventos parem de ocorrer por um tempo. Comum em campos de busca.
Bottleneck
Ponto de estrangulamento no sistema que limita performance geral. Pode ser código lento, query ineficiente ou recurso limitado.
Minification
Processo de remover espaços, comentários e caracteres desnecessários do código para reduzir tamanho de arquivos.
CDN (Content Delivery Network)
Rede de servidores distribuídos geograficamente que entrega conteúdo estático mais rápido ao usuário mais próximo.
Profiling
Análise de performance do código para identificar gargalos, funções lentas e uso excessivo de recursos.
🔒
Segurança
Authentication (Autenticação)
Processo de verificar a identidade do usuário (login com usuário e senha, biometria, etc).
Authorization (Autorização)
Processo de verificar se usuário autenticado tem permissão para acessar determinado recurso ou realizar ação.
JWT (JSON Web Token)
Padrão de token para transmitir informações de forma segura entre cliente e servidor. Usado em autenticação.
Encryption (Criptografia)
Processo de codificar dados para proteger informações sensíveis. Garante que apenas destinatários autorizados possam ler.
SQL Injection
Ataque onde código SQL malicioso é inserido em campos de entrada para manipular ou acessar banco de dados indevidamente.
XSS (Cross-Site Scripting)
Ataque que injeta scripts maliciosos em páginas web visualizadas por outros usuários.
CORS (Cross-Origin Resource Sharing)
Mecanismo de segurança que permite ou bloqueia requisições HTTP entre diferentes domínios.
Environment Variables (.env)
Variáveis de configuração armazenadas fora do código (chaves de API, senhas). Nunca devem ser commitadas no repositório.
📚
Outros Termos Importantes
MVP (Minimum Viable Product)
Versão mínima de um produto com funcionalidades essenciais para validar hipóteses e obter feedback rápido do mercado.
POC (Proof of Concept)
Protótipo ou demonstração para provar viabilidade técnica de uma ideia ou solução antes de investir no desenvolvimento completo.
Legacy Code
Código antigo, geralmente sem documentação adequada e difícil de manter. Funciona mas precisa de modernização.
Downtime
Período em que o sistema está indisponível ou fora do ar. Deve ser minimizado ao máximo.
SLA (Service Level Agreement)
Acordo que define níveis esperados de serviço (uptime de 99.9%, tempo de resposta, etc).
Deprecated
Funcionalidade ou biblioteca marcada como obsoleta. Ainda funciona mas será removida em versões futuras. Deve ser substituída.
Breaking Change
Alteração que quebra compatibilidade com versões anteriores. Requer atualização do código que usa a funcionalidade modificada.
Spike
Pesquisa ou experimento time-boxed para investigar uma solução técnica ou reduzir incertezas antes de implementar.
