Exercício 1
Estrutura do projeto e banco de dados
Sequelize · dotenv · sync()
🎯
Objetivo
Criar a base do projeto: variáveis de ambiente, conexão com o banco via Sequelize e o modelo de usuário. Ao rodar o servidor, a tabela usuarios deve ser criada automaticamente no PostgreSQL.
Tarefa 1.1 Criar o arquivo .env com todas as variáveis
Crie o banco login_sistema no pgAdmin e preencha o .env com suas credenciais:
PORT=3000
DB_HOST=localhost
DB_PORT=5432
DB_NAME=login_sistema
DB_USER=postgres
DB_PASSWORD=sua_senha_aqui
JWT_SECRET=uma_chave_longa_e_segura_aqui_2024!
JWT_EXPIRES_IN=1d
Tarefa 1.2 Criar src/config/database.js — conexão Sequelize
Crie a instância do Sequelize lendo todas as credenciais do process.env. Configure dialect: 'postgres' e logging: false. Exporte como export default.
Tarefa 1.3 Criar src/models/usuario.model.js — modelo Sequelize
Defina o modelo com os campos abaixo usando sequelize.define('Usuario', { ... }):
Campo Tipo Sequelize Restrições
nome DataTypes.STRING allowNull: false
email DataTypes.STRING allowNull: false · unique: true · validate: { isEmail: true }
senha DataTypes.STRING allowNull: false (guarda o hash — nunca a senha original)
ativo DataTypes.BOOLEAN defaultValue: true
Use tableName: 'usuarios' e timestamps: true — o Sequelize criará createdAt e updatedAt automaticamente.
Tarefa 1.4 Criar app.js — servidor com sync()
O app.js deve: (1) ter import 'dotenv/config' na primeira linha, (2) importar Express, (3) importar o Sequelize e o modelo de usuário, (4) chamar sequelize.sync({ alter: true }) antes do listen.
import 'dotenv/config'; // ← SEMPRE PRIMEIRA LINHA
import express from 'express';
import sequelize from './src/config/database.js';
import './src/models/usuario.model.js';

const app = express();
app.use(express.json());

sequelize.sync({ alter: true }).then(() => {
  app.listen(process.env.PORT, () =>
    console.log(`Servidor rodando na porta ${process.env.PORT}`)
  );
});
✔ Como verificar que o Ex 1 está correto
1. Rode node app.js — o terminal deve exibir "Servidor rodando na porta 3000" sem erros.
2. Abra o pgAdmin e confira que a tabela usuarios foi criada automaticamente com as colunas id, nome, email, senha, ativo, createdAt, updatedAt.
3. Pare o servidor, altere o nome de um campo no model e rode novamente — o alter: true deve ajustar a tabela automaticamente.
Exercício 2
Rota de cadastro com criptografia de senha
bcryptjs · POST /auth/cadastro
🎯
Objetivo
Criar a rota de cadastro que recebe nome, email e senha, gera um hash da senha com Bcrypt antes de salvar e retorna os dados do usuário criado sem expor a senha. O campo senha nunca deve aparecer na resposta.
Tarefa 2.1 Criar src/controllers/auth.controller.js
Implemente o handler cadastrar seguindo este fluxo:
1
Destruture { nome, email, senha } do req.body. Valide se os três campos foram enviados — retorne 400 se faltar algum.
2
Verifique se já existe um usuário com o mesmo email usando Usuario.findOne({ where: { email } }). Se existir, retorne 409 com { erro: 'E-mail já cadastrado' }.
3
Gere o hash: const senhaHash = await bcrypt.hash(senha, 10). Nunca salve a senha em texto puro.
4
Salve com Usuario.create({ nome, email, senha: senhaHash }). Retorne 201 com { id, nome, email, createdAt } — sem o campo senha.
Tarefa 2.2 Criar src/routes/auth.routes.js e registrar no app.js
Crie o Router com a rota de cadastro e registre-o no app.js com o prefixo /auth.
POST  /auth/cadastro  → pública — cadastra usuário com senha hasheada (retorna 201)
Body da requisição (enviar)
{
  "nome": "Ana Silva",
  "email": "ana@email.com",
  "senha": "minhasenha123"
}
Resposta esperada (201)
{
  "id": 1,
  "nome": "Ana Silva",
  "email": "ana@email.com",
  "createdAt": "2024-..."
}
✔ Como verificar que o Ex 2 está correto
1. POST em /auth/cadastro — deve retornar 201. A resposta não deve conter o campo senha.
2. No pgAdmin, confira que a coluna senha contém um hash começando com $2b$ — nunca a senha original.
3. Tente cadastrar o mesmo email duas vezes — deve retornar 409.
4. Envie o body sem algum campo — deve retornar 400.
Exercício 3
Rota de login e geração do token JWT
bcryptjs · jwt.sign() · POST /auth/login
🎯
Objetivo
Criar a rota de login que valida o email e a senha do usuário com Bcrypt e, se corretos, retorna um token JWT assinado. Este token será usado pelo front-end para autenticar todas as próximas requisições.
Tarefa 3.1 Adicionar o handler login no auth.controller.js
Implemente o handler login seguindo este fluxo:
1
Receba { email, senha } do body. Valide os dois campos — retorne 400 se faltar.
2
Busque o usuário com Usuario.findOne({ where: { email } }). Se não existir, retorne 401 com { erro: 'Credenciais inválidas' }.
3
Valide a senha com await bcrypt.compare(senha, usuario.senha). Se inválida, retorne 401 com a mesma mensagem genérica — não diga se foi o email ou a senha que errou.
4
Gere o token: jwt.sign({ id: usuario.id, nome: usuario.nome, email: usuario.email }, process.env.JWT_SECRET, { expiresIn: process.env.JWT_EXPIRES_IN }).
5
Retorne 200 com { token, usuario: { id, nome, email } }. O front-end vai armazenar o token e usá-lo nas próximas requisições.
Tarefa 3.2 Adicionar a rota de login ao auth.routes.js
POST  /auth/cadastro  → pública — cadastra usuário (Ex 2)
POST  /auth/login     → pública — valida credenciais e retorna token JWT (200)
Body da requisição (enviar)
{
  "email": "ana@email.com",
  "senha": "minhasenha123"
}
Resposta esperada (200) — o front-end recebe
{
  "token": "eyJhbGci...",
  "usuario": {
    "id": 1, "nome": "Ana",
    "email": "ana@email.com"
  }
}
✔ Como verificar que o Ex 3 está correto
1. POST em /auth/login com credenciais corretas — deve retornar 200 com o token.
2. Cole o token em jwt.io — confirme que o payload contém id, nome e email.
3. Tente com senha errada — deve retornar 401 com "Credenciais inválidas" (sem dizer qual campo errou).
4. Tente com email que não existe — deve retornar 401 com a mesma mensagem do passo 3.
Exercício 4
Rotas protegidas, perfil e preparação para o front-end
Middleware · CORS · PUT · DELETE
🎯
Objetivo
Proteger rotas com middleware JWT, criar endpoints de perfil e gerenciamento de conta, e configurar o CORS para que o front-end consiga se comunicar com o back-end sem erros de bloqueio de origem.
Tarefa 4.1 Criar src/middlewares/auth.middleware.js
Crie a função autenticar(req, res, next):
1
Leia o cabeçalho Authorization. Se ausente ou não iniciar com Bearer , retorne 401.
2
Extraia o token e verifique com jwt.verify(token, process.env.JWT_SECRET). Se inválido ou expirado, retorne 401.
3
Se válido, salve os dados decodificados em req.usuario e chame next().
Tarefa 4.2 Criar src/controllers/usuario.controller.js e src/routes/usuario.routes.js
Implemente os handlers abaixo. Todos usam req.usuario.id (do token) para identificar o usuário — nunca aceite o id via URL ou body sem validação:
GET
/usuario/perfil — retorna os dados do usuário logado (sem a senha). Use Usuario.findByPk(req.usuario.id, { attributes: { exclude: ['senha'] } }).
PUT
/usuario/perfil — atualiza nome e/ou email do usuário logado. Se enviar nova senha, gere o hash antes de salvar. Retorne o usuário atualizado (sem a senha).
DELETE
/usuario/conta — desativa a conta do usuário logado (não deleta do banco — muda ativo = false). Retorne 204.
Todas as rotas de usuário são privadas — aplique o middleware:
router.get('/perfil', autenticar, C.perfil);
router.put('/perfil', autenticar, C.atualizarPerfil);
router.delete('/conta', autenticar, C.desativarConta);
Tarefa 4.3 Configurar CORS no app.js — essencial para o front-end
Sem CORS configurado, o navegador bloqueará todas as requisições do front-end para o back-end. Adicione no app.js antes de qualquer rota:
import cors from 'cors';

// Permite qualquer origem em desenvolvimento
// Em produção, substitua '*' pela URL do front-end
app.use(cors({
  origin: '*',
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization']
}));
app.use(express.json());
Mapa completo de rotas da API
── Públicas (não exigem token) ──────────────────────────────
POST  /auth/cadastro         → cadastra com senha hasheada (201)
POST  /auth/login            → retorna token JWT (200)
── Privadas (exigem Bearer token no cabeçalho) ──────────────
GET    /usuario/perfil        → dados do usuário logado (sem senha)
PUT    /usuario/perfil        → atualiza nome, email ou senha
DELETE /usuario/conta         → desativa conta (ativo = false, retorna 204)
✔ Como verificar que o Ex 4 está correto — sequência completa no Postman ou Insomnia
1. POST /auth/cadastro — crie uma conta.
2. POST /auth/login — faça login e copie o token.
3. GET /usuario/perfil sem o token — deve retornar 401.
4. GET /usuario/perfil com Authorization: Bearer <token> — deve retornar os dados do usuário (sem a senha).
5. PUT /usuario/perfil com { "nome": "Ana Costa" } — deve atualizar e retornar os novos dados.
6. DELETE /usuario/conta — deve retornar 204. No pgAdmin, confirme que ativo = false.
7. Tente fazer login com a conta desativada — implemente a verificação de ativo === false no handler de login retornando 403 com "Conta desativada".

// Checklist de status codes que a API deve retornar corretamente:
200 → login bem-sucedido, perfil retornado, atualização concluída
201 → cadastro realizado com sucesso
204 → conta desativada (sem corpo de resposta)
400 → campos obrigatórios ausentes no body
401 → credenciais inválidas ou token ausente/expirado
403 → conta desativada tentando logar
409 → e-mail já cadastrado
500 → erro interno (não deve aparecer se o código estiver correto)
