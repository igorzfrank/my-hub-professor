Exercício 1
API de Autenticação — Cadastro e Login com JWT
POST · JWT
Você irá criar uma API de autenticação completa. O usuário se cadastra com nome, e-mail e senha — a senha é armazenada como hash gerado pelo Bcrypt. No login, a senha é comparada ao hash e, se válida, um JWT é retornado para uso futuro.
Passo 1 Banco de dados e tabela — executar no pgAdmin
SQL — criação
-- 1. Crie o banco
CREATE DATABASE auth_app;

-- 2. Conecte ao banco e execute o restante

CREATE TABLE usuarios (
  id SERIAL PRIMARY KEY,
  nome TEXT NOT NULL,
  email TEXT UNIQUE NOT NULL,
  senha_hash TEXT NOT NULL,
  criado_em TIMESTAMP DEFAULT NOW()
);
Passo 2 Arquivo .env
.env — variáveis de ambiente
DB_HOST=localhost
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=sua_senha
DB_NAME=auth_app
JWT_SECRET=segredo_super_secreto_troque_em_producao
JWT_EXPIRES_IN=1h
Passo 3 O que você deve construir
db.js — conexão via variáveis de ambiente
Crie o Pool usando process.env.DB_HOST, process.env.DB_USER etc. Nunca coloque credenciais diretamente no código.
usuario.model.js — queries
Implemente criarUsuario(nome, email, senhaHash) e buscarPorEmail(email). Use RETURNING id, nome, email no INSERT — nunca retorne o hash.
auth.controller.js — cadastrar e login
cadastrar: use bcrypt.hash(senha, 10) antes de salvar. Retorne 201 com os dados do usuário (sem o hash).
login: busque o usuário, use bcrypt.compare(senha, hash). Se válido, gere o token com jwt.sign({ id, email }, JWT_SECRET, { expiresIn }). Retorne 401 se as credenciais forem inválidas.
auth.routes.js + app.js
Crie o Router com as duas rotas e registre-o no app.js com o prefixo /auth.
Rotas esperadas
POST /auth/register → cria usuário com senha em hash (retorna 201)
POST /auth/login → valida credenciais e retorna JWT (401 se inválido)
Body — POST /auth/register
{
  "nome": "Ana Lima",
  "email": "ana@email.com",
  "senha": "minhasenha123"
}
Body — POST /auth/login
{
  "email": "ana@email.com",
  "senha": "minhasenha123"
}
Resposta esperada no login (200 OK)
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "usuario": {
    "id": 1,
    "nome": "Ana Lima",
    "email": "ana@email.com"
  }
}
Como verificar
1. Faça um POST em /auth/register — deve retornar 201 com os dados do usuário (sem a senha).
2. Confirme no pgAdmin que a coluna senha_hash contém um hash Bcrypt (começa com $2b$).
3. Faça um POST em /auth/login — deve retornar o token JWT.
4. Tente logar com a senha errada — deve retornar 401.
Exercício 2
Middleware de Autenticação — Rotas Protegidas com JWT
MIDDLEWARE · BEARER
Agora que você sabe gerar um JWT, aprenda a proteger rotas com ele. Você criará um middleware que valida o token enviado no header Authorization e bloqueia requisições não autenticadas antes que cheguem ao controller.
Passo 1 Banco de dados — use o mesmo banco do Exercício 1
SQL — nova tabela no banco auth_app
-- Conecte ao banco auth_app e execute

CREATE TABLE notas (
  id SERIAL PRIMARY KEY,
  usuario_id INTEGER NOT NULL REFERENCES usuarios(id) ON DELETE CASCADE,
  titulo TEXT NOT NULL,
  conteudo TEXT,
  criado_em TIMESTAMP DEFAULT NOW()
);
Passo 2 O que você deve construir
auth.middleware.js — verificação do token
Extraia o token do header Authorization: Bearer <token>. Use jwt.verify(token, JWT_SECRET) — se válido, salve o payload decodificado em req.usuario e chame next(). Se inválido ou ausente, retorne 401.
nota.model.js — queries com filtro por usuário
Implemente buscarPorUsuario(usuarioId) e criar(usuarioId, titulo, conteudo). Sempre filtre por usuario_id — cada usuário só acessa suas próprias notas.
nota.controller.js — usa req.usuario injetado pelo middleware
Os handlers listar e criar devem pegar o id do usuário logado a partir de req.usuario.id — que foi preenchido pelo middleware.
nota.routes.js — aplique o middleware nas rotas
Importe o middleware e adicione-o como segundo argumento nas rotas protegidas: router.get('/', autenticar, listar). Registre no app.js com o prefixo /notas.
Rotas esperadas
GET /notas → 🔒 lista as notas do usuário autenticado (401 sem token)
POST /notas → 🔒 cria uma nota vinculada ao usuário do token (401 sem token)
Header obrigatório nas requisições
Authorization: Bearer eyJhbGci...
Body — POST /notas
{
  "titulo": "Minha primeira nota",
  "conteudo": "Conteúdo da nota aqui"
}
Como verificar
1. Faça login em /auth/login e copie o token retornado.
2. Tente GET /notas sem o header — deve retornar 401.
3. Tente GET /notas com o token no header — deve retornar a lista (vazia no início).
4. Crie notas com POST e confirme que o GET lista apenas as do usuário logado.
Exercício 3
Troca de Senha — Re-hash com Bcrypt e Invalidação de Token
BCRYPT · UPDATE · AUTH
Neste exercício você implementará a troca de senha de forma segura: o usuário autenticado informa a senha atual, que é validada com bcrypt.compare, e a nova senha é armazenada como um novo hash. Ao final, um novo token é emitido.
Passo 1 Banco de dados — use o mesmo banco auth_app
SQL — nenhuma tabela nova necessária
-- Adicione a coluna updated_at para rastrear a última atualização

ALTER TABLE usuarios
  ADD COLUMN updated_at TIMESTAMP DEFAULT NOW();

-- Função para atualizar updated_at automaticamente
CREATE OR REPLACE FUNCTION atualizar_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_usuarios_updated_at
BEFORE UPDATE ON usuarios
FOR EACH ROW EXECUTE FUNCTION atualizar_updated_at();
Passo 2 O que você deve construir
usuario.model.js — novas queries
Adicione buscarPorId(id) (retorna o hash também, para validação interna) e atualizarSenha(id, novoHash). Use RETURNING id, nome, email, updated_at.
usuario.controller.js — handler trocarSenha
Use req.usuario.id (do middleware) para buscar o usuário. Valide a senhaAtual com bcrypt.compare — retorne 400 se incorreta. Gere um novo hash com bcrypt.hash(novaSenha, 10), salve no banco e emita um novo JWT.
usuario.routes.js — rota protegida de troca de senha
Adicione a rota PATCH /usuarios/senha com o middleware autenticar aplicado. Registre o router no app.js com prefixo /usuarios.
Regras importantes

• A rota deve ser protegida pelo middleware JWT
• Valide a senha atual antes de permitir a troca
• Nunca salve a senha como texto — sempre use bcrypt.hash
• Retorne um novo token após a troca — o anterior continua válido até expirar (comportamento esperado para fins didáticos)
• Retorne 400 se a senha atual estiver incorreta e 404 se o usuário não existir
Rota esperada
PATCH /usuarios/senha → 🔒 troca a senha do usuário autenticado e retorna novo JWT
Body — PATCH /usuarios/senha
{
  "senhaAtual": "minhasenha123",
  "novaSenha": "senhaForte@2025"
}
Resposta esperada (200 OK)
{
  "mensagem": "Senha atualizada",
  "token": "eyJhbGci..."
}
Como verificar
1. Faça login e copie o token.
2. Envie PATCH /usuarios/senha com o token e as senhas — deve retornar novo token.
3. Tente logar com a senha antiga — deve retornar 401.
4. Tente logar com a nova senha — deve funcionar.
5. Confirme no pgAdmin que o campo updated_at foi atualizado e que o hash mudou.
