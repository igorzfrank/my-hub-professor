Módulo 1
Autenticação vs. Autorização — qual é a diferença?
1.1 — Dois conceitos distintos e complementares
Na aula de JWT aprendemos a verificar quem é o usuário que faz a requisição. Mas saber a identidade não é suficiente — um sistema real precisa saber também o que aquele usuário tem permissão de fazer. Esses são dois problemas diferentes que precisam de soluções diferentes.
🪪
Autenticação
Quem é você?
Verificar que o usuário é quem diz ser. Feita no login com e-mail e senha, gerando um token JWT.
"Você é o Carlos. Token válido."
🛡️
Autorização
O que você pode fazer?
Verificar se o usuário autenticado tem permissão para acessar aquele recurso ou executar aquela ação.
"Carlos é leitor. Não pode criar posts."
Analogia: em um hospital, o crachá de funcionário é a autenticação — prova que você trabalha ali. Mas um enfermeiro e um médico têm crachás diferentes com permissões diferentes: o enfermeiro pode fazer curativos, mas não pode prescrever medicamentos. Isso é autorização.
1.2 — O que são Roles (papéis)?
Uma role (ou papel) é uma etiqueta que classifica o tipo de usuário e define o conjunto de ações que ele pode realizar. Em vez de dar permissões individuais a cada usuário, atribui-se uma role — e a role carrega as permissões automaticamente.
Role Ler dados Criar registros Editar qualquer registro Excluir usuários
leitor ✓ ✗ ✗ ✗
editor ✓ ✓ ✓ ✗
admin ✓ ✓ ✓ ✓
Módulo 2
Armazenando e transportando a Role
2.1 — A role no banco de dados
A role é uma coluna como qualquer outra na tabela de usuários. Definimos um valor padrão (DEFAULT 'leitor') para que qualquer usuário cadastrado comece com o perfil mais restrito automaticamente.
SQL — tabela de usuários com role
CREATE TABLE usuarios (
  id         SERIAL PRIMARY KEY,
  nome       TEXT NOT NULL,
  email      TEXT UNIQUE NOT NULL,
  senha_hash  TEXT NOT NULL,
  role       TEXT NOT NULL DEFAULT 'leitor' -- 'leitor' | 'editor' | 'admin'
);
2.2 — Incluindo a role no payload do JWT
Ao fazer login com sucesso, a role do usuário é incluída no payload do token. Isso é fundamental: o middleware de autorização vai ler a role direto do token, sem precisar consultar o banco a cada requisição.
usuario.controller.js — handler de login
export async function login(req, res) {
  try {
    const { email, senha } = req.body;
    const usuario = await UsuarioModel.buscarPorEmail(email);
    if (!usuario) return res.status(401).json({ erro: 'Credenciais inválidas' });

    const senhaValida = await bcrypt.compare(senha, usuario.senha_hash);
    if (!senhaValida) return res.status(401).json({ erro: 'Credenciais inválidas' });

    // ✅ role incluída no payload — disponível em req.usuario.role
    const token = jwt.sign(
      { id: usuario.id, nome: usuario.nome, role: usuario.role },
      process.env.JWT_SECRET,
      { expiresIn: process.env.JWT_EXPIRES_IN }
    );
    res.json({ token });
  } catch { res.status(500).json({ erro: 'Erro interno' }); }
}
2.3 — O que fica dentro do token
Após o login, o payload decodificado do token contém a role junto com os outros dados. O middleware de autorização vai ler esse campo diretamente de req.usuario.role:
Payload de um leitor
{
  "id": 3,
  "nome": "Carla",
  "role": "leitor",
  "iat": 1715000000,
  "exp": 1715086400
}
Payload de um admin
{
  "id": 1,
  "nome": "Rafael",
  "role": "admin",
  "iat": 1715000000,
  "exp": 1715086400
}
Segurança — payload não é segredo
O payload do JWT é codificado em Base64, não criptografado — qualquer pessoa que tiver o token pode ler o payload em jwt.io. Por isso, jamais inclua senhas, dados bancários ou informações sensíveis no payload. A assinatura garante que o conteúdo não foi adulterado, mas não que está oculto.
Módulo 3
Middlewares de Autorização
3.1 — A cadeia de middlewares
A autorização sempre vem depois da autenticação. O middleware de autenticação verifica o token e preenche req.usuario. Só então o middleware de autorização pode ler req.usuario.role.
Requisição → [ autenticar ] → [ exigirRole(...) ] → Handler da rota → Resposta
3.2 — Middleware de autenticação (revisão)
src/middlewares/auth.middleware.js
import jwt from 'jsonwebtoken';

// ── Verifica se o token é válido ────────────────────
export function autenticar(req, res, next) {
  const authHeader = req.headers['authorization'];
  if (!authHeader?.startsWith('Bearer '))
    return res.status(401).json({ erro: 'Token não fornecido' });

  const token = authHeader.split(' ')[1];
  try {
    req.usuario = jwt.verify(token, process.env.JWT_SECRET);
    next();
  } catch {
    res.status(401).json({ erro: 'Token inválido ou expirado' });
  }
}
3.3 — Middleware de autorização por role
O middleware de autorização é uma factory function — uma função que retorna outra função. Isso permite passar quais roles são aceitas diretamente na definição da rota, tornando o código flexível e reutilizável.
src/middlewares/auth.middleware.js — adicionar abaixo do autenticar
// ── Verifica se o usuário tem uma das roles permitidas ──
// Uso: exigirRole('admin') ou exigirRole('admin', 'editor')
export function exigirRole(...rolesPermitidas) {
  return function(req, res, next) {
    const roleDoUsuario = req.usuario?.role;

    if (!rolesPermitidas.includes(roleDoUsuario)) {
      return res.status(403).json({
        erro: 'Acesso negado. Permissão insuficiente.'
      });
    }
    next();
  };
}
401 vs. 403 — qual a diferença?
401 Unauthorized — o usuário não está autenticado (não tem token ou o token é inválido). É como tentar entrar em um clube sem apresentar o crachá.

403 Forbidden — o usuário está autenticado mas não tem permissão para aquela ação. É como ter o crachá de visitante e tentar entrar na sala restrita de funcionários.
3.4 — Aplicando os middlewares nas rotas
Com a factory function exigirRole(), a definição das rotas fica expressiva e fácil de entender: qualquer desenvolvedor que leia o router sabe imediatamente quem pode acessar cada rota.
src/routes/post.routes.js
import { Router } from 'express';
import * as C from '../controllers/post.controller.js';
import { autenticar, exigirRole } from '../middlewares/auth.middleware.js';

const router = Router();

// Pública — qualquer pessoa pode ler
router.get('/', C.listar);

// Privada — qualquer usuário logado pode ler um post
router.get('/:id', autenticar, C.buscarPorId);

// Restrita — apenas editor ou admin podem criar
router.post('/', autenticar, exigirRole('editor', 'admin'), C.criar);

// Restrita — apenas editor ou admin podem editar
router.put('/:id', autenticar, exigirRole('editor', 'admin'), C.atualizar);

// Restrita — apenas admin pode excluir
router.delete('/:id', autenticar, exigirRole('admin'), C.remover);

export default router;
Módulo 4
Boas práticas de autorização
4.1 — Princípio do menor privilégio
Todo usuário deve começar com a role mais restrita possível e receber mais permissões apenas quando necessário. Isso minimiza o impacto de um eventual comprometimento de conta.
❌ Errado
Dar role admin para todo novo usuário e remover permissões depois.
✅ Correto
Todo novo usuário começa como leitor por padrão (DEFAULT 'leitor' no banco). O admin promove manualmente quando necessário.
4.2 — Nunca confie apenas no frontend
Esconder um botão de "Excluir" no frontend para usuários sem permissão é uma boa experiência de uso — mas não é segurança. Alguém pode fazer a requisição diretamente pelo Postman. A autorização deve sempre ser verificada no backend, a cada requisição.
❌ Inseguro
Só ocultar o botão de exclusão no HTML. A rota DELETE /posts/1 ainda pode ser chamada diretamente.
✅ Seguro
Ocultar no frontend e proteger a rota com autenticar + exigirRole('admin') no backend.
4.3 — Rota para promoção de role (exclusiva do admin)
Uma API bem construída deve ter uma rota específica para que um admin possa alterar a role de outro usuário. Essa rota deve ser protegida com exigirRole('admin').
Exemplo — rota para promover usuário
// Na rota: PATCH /usuarios/:id/role — apenas admin
router.patch(
  '/:id/role',
  autenticar,
  exigirRole('admin'),
  C.atualizarRole
);

// No controller:
export async function atualizarRole(req, res) {
  try {
    const { role } = req.body; // 'leitor' | 'editor' | 'admin'
    const rolesValidas = ['leitor', 'editor', 'admin'];
    if (!rolesValidas.includes(role))
      return res.status(400).json({ erro: 'Role inválida' });
    const usuario = await UsuarioModel.atualizarRole(req.params.id, role);
    if (!usuario) return res.status(404).json({ erro: 'Usuário não encontrado' });
    res.json({ mensagem: 'Role atualizada com sucesso', usuario });
  } catch { res.status(500).json({ erro: 'Erro interno' }); }
}

import { Router } from 'express';
import * as C from '../controllers/post.controller.js';
import { autenticar, exigirRole } from '../middlewares/auth.middleware.js';

const router = Router();

// ── Públicas ──────────────────────────────────────
router.get('/', C.listar);                  // todos

// ── Exige login (qualquer role) ────────────────────
router.get('/:id', autenticar, C.buscarPorId); // logado

// ── Exige editor ou admin ──────────────────────────
router.post ('/', autenticar, exigirRole('editor','admin'), C.criar);
router.put ('/:id', autenticar, exigirRole('editor','admin'), C.atualizar);

// ── Exige admin ────────────────────────────────────
router.delete('/:id', autenticar, exigirRole('admin'), C.remover);

export default router;

Sequência de testes no Postman ou Insomnia
1. POST /usuarios/cadastro — crie um usuário leitor e um admin ("role": "admin" no body).
2. Faça login com o admin. Copie o token e cole em jwt.io — verifique que "role": "admin" aparece no payload.
3. Com o token do admin, POST /posts — deve retornar 201.
4. Faça login com o leitor. Com o token do leitor, POST /posts — deve retornar 403.
5. Com o token do admin, PATCH /usuarios/:id/role com { "role": "editor" } — promova o leitor a editor.
6. Faça login novamente com o usuário promovido. Com o novo token, POST /posts — agora deve retornar 201.
