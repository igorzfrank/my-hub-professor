Módulo 1
O problema — senhas e acesso sem proteção
1.1 — O que acontece em uma API sem autenticação
Nas aulas anteriores, qualquer pessoa que soubesse o endereço da API conseguia listar, criar, editar e deletar dados livremente. Isso é aceitável em um projeto de aprendizagem, mas em um sistema real é uma falha grave. Uma API de produção precisa responder a duas perguntas antes de processar qualquer requisição:
🔑
Autenticação
Quem é você?
Verificar que o usuário é quem diz ser — geralmente com e-mail e senha.
🛡️
Autorização
O que você pode fazer?
Verificar se o usuário autenticado tem permissão para acessar aquele recurso.
1.2 — Dois problemas, duas soluções
Problema 1 — Armazenar senhas
Se o banco de dados vazar e as senhas estiverem em texto puro, todos os usuários ficam comprometidos. Senhas nunca podem ser salvas como estão.
senha: "minhasenha123" ← ERRADO
Problema 2 — Controlar acesso
Depois de fazer login, como a API sabe que as próximas requisições vêm do mesmo usuário autenticado? HTTP é sem estado — cada requisição é independente.
GET /meus-dados ← quem está pedindo?
A solução: usamos o Bcrypt para transformar senhas em hashes seguros antes de salvar no banco, e o JWT para gerar um token que o cliente envia em cada requisição, provando que está autenticado.
Módulo 2
Bcrypt — protegendo senhas com hash
2.1 — O que é um hash?
Um hash é uma transformação de mão única: você transforma a senha em uma sequência de caracteres embaralhada. O importante é que não existe caminho de volta — dado o hash, é impossível recuperar a senha original. Para verificar se uma senha está correta, você gera o hash dela novamente e compara com o que está salvo.
Exemplo de hash gerado pelo Bcrypt
// Senha original (nunca salva no banco)
"minhasenha123"

// Hash gerado pelo Bcrypt (o que é salvo no banco)
"$2b$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy"
Analogia: o hash é como passar carne pelo moedor. Você não consegue pegar o resultado e desmontar para recuperar o pedaço original. Para saber se dois pedaços eram iguais, você passa os dois pelo mesmo moedor e compara o resultado.
2.2 — O que é o Salt e o Cost Factor?
O Bcrypt adiciona dois conceitos que tornam o hash ainda mais seguro:
Salt — valor aleatório único
Antes de gerar o hash, o Bcrypt adiciona um trecho aleatório à senha — o salt. Isso garante que dois usuários com a mesma senha terão hashes completamente diferentes, tornando impossível detectar usuários com senhas iguais olhando para o banco.
Cost Factor (rounds) — dificuldade intencional
O parâmetro saltRounds define quantas vezes o algoritmo repete a operação. Um valor de 10 significa 2¹⁰ = 1024 repetições. Isso torna o processo intencionalmente lento — o suficiente para ser imperceptível para um usuário, mas devastador para um ataque de força bruta. O valor padrão recomendado é 10.
2.3 — Instalação e as duas funções principais
Terminal
npm install bcryptjs
hash() — gerar o hash (cadastro)
import bcrypt from 'bcryptjs';

const senha = 'minhasenha123';
const saltRounds = 10;

const hash = await bcrypt.hash(senha, saltRounds);
// hash → "$2b$10$N9qo8u..."
// Salve o hash no banco, não a senha!
compare() — verificar (login)
import bcrypt from 'bcryptjs';

const senhaDigitada = 'minhasenha123';
const hashDoBanco = '$2b$10$N9qo...';

const valida = await bcrypt.compare(senhaDigitada, hashDoBanco);
// valida → true ou false
Regra fundamental
Nunca salve a senha original no banco. Sempre gere o hash com bcrypt.hash() antes de inserir. No login, use bcrypt.compare() para validar — jamais compare strings diretamente.
Módulo 3
JWT — controlando o acesso com tokens
3.1 — O que é JWT?
JWT significa JSON Web Token. É uma string compacta que o servidor gera após um login bem-sucedido e entrega ao cliente. Em toda requisição seguinte, o cliente envia esse token no cabeçalho HTTP — e o servidor o valida sem precisar consultar o banco de dados.
Analogia: o JWT é como o crachá de acesso de uma empresa. Quando você entra, passa na recepção (faz login), apresenta seu documento e recebe um crachá. Nos dias seguintes, não precisa mais passar pela recepção — basta mostrar o crachá nas catracas (rotas protegidas). O crachá tem prazo de validade e é difícil de falsificar.
3.2 — Estrutura do token JWT
Um token JWT é formado por três partes separadas por ponto, cada uma codificada em Base64:
Exemplo de token JWT
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpZCI6MSwibm9tZSI6IkNhcmxvcyIsImlhdCI6MTcxNTAwMDAwMH0.dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk
Header
Algoritmo usado para assinar o token. Geralmente HS256.
Payload
Os dados do usuário: id, nome, validade. Não coloque senhas aqui — o payload é legível!
Signature
Assinatura digital gerada com uma chave secreta. Garante que o token não foi adulterado.
3.3 — Instalação e as duas funções principais
Terminal
npm install jsonwebtoken
sign() — gerar o token (login)
import jwt from 'jsonwebtoken';

const payload = { id: 1, nome: 'Carlos' };
const segredo = process.env.JWT_SECRET;
const opcoes = { expiresIn: '1d' };

const token = jwt.sign(payload, segredo, opcoes);
// Envie o token para o cliente
verify() — validar o token (rotas protegidas)
import jwt from 'jsonwebtoken';

try {
  const dados = jwt.verify(token, process.env.JWT_SECRET);
  // dados → { id: 1, nome: 'Carlos', iat: ..., exp: ... }
  // Token válido — prossiga com a requisição
} catch (erro) {
  // Token inválido ou expirado
}
3.4 — A chave secreta no .env
A chave secreta (JWT_SECRET) é o que garante a autenticidade da assinatura. Se ela vazar, qualquer pessoa pode fabricar tokens válidos. Ela deve ser uma string longa e aleatória, armazenada exclusivamente no .env.
.env — adicionar a chave secreta
# ... variáveis do banco ...
JWT_SECRET=uma_string_muito_longa_e_aleatoria_aqui_2026!
JWT_EXPIRES_IN=1d
Módulo 4
Middleware de autenticação — protegendo rotas
4.1 — O que é um middleware?
Um middleware é uma função que fica no meio do caminho entre a requisição e o handler final da rota. Ele recebe req, res e uma terceira função chamada next. Se tudo estiver certo, chama next() para continuar. Se não, interrompe e responde com erro.
Requisição → [ Middleware JWT ] → Handler da rota → Resposta
4.2 — Como o cliente envia o token
O cliente deve enviar o token no cabeçalho Authorization de cada requisição, no formato Bearer <token>. Esse é o padrão da indústria.
Cabeçalho HTTP enviado pelo cliente (Postman / Insomnia)
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
4.3 — O middleware de autenticação
src/middlewares/auth.middleware.js
import jwt from 'jsonwebtoken';

export function autenticar(req, res, next) {
  const authHeader = req.headers['authorization'];

  // Verifica se o cabeçalho existe e começa com "Bearer "
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ erro: 'Token não fornecido' });
  }

  // Extrai o token removendo o prefixo "Bearer "
  const token = authHeader.split(' ')[1];

  try {
    const dados = jwt.verify(token, process.env.JWT_SECRET);
    req.usuario = dados; // disponibiliza os dados do token no req
    next(); // token válido — passa para o handler da rota
  } catch {
    res.status(401).json({ erro: 'Token inválido ou expirado' });
  }
}
4.4 — Aplicando o middleware nas rotas
Basta passar o middleware como segundo argumento na rota. Rotas públicas (como cadastro e login) ficam sem o middleware; rotas privadas recebem autenticar como guardião.
src/routes/usuario.routes.js
import { Router } from 'express';
import * as C from '../controllers/usuario.controller.js';
import { autenticar } from '../middlewares/auth.middleware.js';

const router = Router();

// Rotas públicas — sem middleware
router.post('/cadastro', C.cadastrar);
router.post('/login', C.login);

// Rotas privadas — exigem token válido
router.get('/perfil', autenticar, C.perfil);
router.put('/perfil', autenticar, C.atualizar);

export default router;

2 / 3 src/controllers/usuario.controller.js
import bcrypt from 'bcryptjs';
import jwt from 'jsonwebtoken';
import * as UsuarioModel from '../models/usuario.model.js';

// ── Cadastro ──────────────────────────────────────────
export async function cadastrar(req, res) {
  try {
    const { nome, email, senha } = req.body;
    if (!nome || !email || !senha)
      return res.status(400).json({ erro: 'nome, email e senha são obrigatórios' });

    // Gera o hash antes de salvar — NUNCA salve a senha direta
    const senhaHash = await bcrypt.hash(senha, 10);
    const usuario = await UsuarioModel.criar(nome, email, senhaHash);
    res.status(201).json(usuario);
  } catch {
    res.status(500).json({ erro: 'Erro ao cadastrar usuário' });
  }
}

// ── Login ─────────────────────────────────────────────
export async function login(req, res) {
  try {
    const { email, senha } = req.body;

    const usuario = await UsuarioModel.buscarPorEmail(email);
    if (!usuario)
      return res.status(401).json({ erro: 'Credenciais inválidas' });

    // Compara a senha digitada com o hash do banco
    const senhaValida = await bcrypt.compare(senha, usuario.senha_hash);
    if (!senhaValida)
      return res.status(401).json({ erro: 'Credenciais inválidas' });

    // Gera o token JWT com os dados do usuário no payload
    const token = jwt.sign(
      { id: usuario.id, nome: usuario.nome },
      process.env.JWT_SECRET,
      { expiresIn: process.env.JWT_EXPIRES_IN }
    );

    res.json({ token });
  } catch {
    res.status(500).json({ erro: 'Erro ao fazer login' });
  }
}

// ── Perfil (rota protegida) ───────────────────────────
export async function perfil(req, res) {
  // req.usuario foi preenchido pelo middleware com os dados do token
  res.json({ mensagem: 'Acesso autorizado', usuario: req.usuario });
}

Resumo — o papel de cada peça
Ferramenta Quando usar Função
bcrypt.hash() Cadastro de usuário Transforma a senha em hash antes de salvar no banco
bcrypt.compare() Login Verifica se a senha digitada corresponde ao hash do banco
jwt.sign() Login bem-sucedido Gera o token com os dados do usuário e o entrega ao cliente
jwt.verify() Middleware — toda rota protegida Valida o token enviado no cabeçalho e libera ou bloqueia o acesso
