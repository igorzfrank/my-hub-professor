Design Patterns — Padrões de Projeto de Software
Nesta aula você aprenderá o que são Design Patterns, por que eles existem e como aplicá-los em JavaScript para escrever código mais organizado, seguro, reutilizável e de fácil manutenção. Cada padrão será apresentado com exemplos comparativos: primeiro o problema sem o padrão, depois a solução com ele.
Capacidades Trabalhadas
 
Reconhecer requisitos de qualidade, integridade, usabilidade e segurança da informação no desenvolvimento de sistemas.
O que são Design Patterns?
Um Design Pattern (Padrão de Projeto) é uma solução reutilizável para um problema recorrente no desenvolvimento de software. Não é um código pronto para copiar — é um modelo conceitual que pode ser adaptado a diferentes linguagens e contextos.
O conceito foi formalizado em 1994 pelo livro Design Patterns: Elements of Reusable Object-Oriented Software, escrito por Erich Gamma, Richard Helm, Ralph Johnson e John Vlissides — conhecidos na comunidade como Gang of Four (GoF). Eles catalogaram 23 padrões agrupados em três categorias: Criacionais, Estruturais e Comportamentais.
Analogia do mundo real: imagine que um engenheiro civil está projetando a estrutura de um prédio. Ele não inventa do zero como calcular vigas e pilares — ele aplica padrões consagrados da engenharia que já foram testados em milhares de construções. Design Patterns fazem o mesmo para o software: são soluções que já provaram funcionar, evitando que cada desenvolvedor "reinvente a roda" a cada novo projeto.
Por que usar Design Patterns?
Código sem padrões cresce de forma desorganizada. Com o tempo, torna-se difícil de entender, modificar e testar — fenômeno chamado de código espaguete. Os Design Patterns atacam esse problema diretamente, oferecendo quatro benefícios fundamentais:
Qualidade
O código fica mais legível, previsível e alinhado com boas práticas reconhecidas pela comunidade. Novos desenvolvedores conseguem entender o sistema mais rapidamente.
Integridade
Padrões como Singleton e Factory controlam a criação de objetos, evitando instâncias duplicadas, estados inconsistentes e comportamentos imprevisíveis que comprometem os dados.
Usabilidade do Código
Padrões como Observer e Strategy tornam o código mais flexível e extensível. Adicionar novas funcionalidades não exige alterar o que já funciona — apenas estender.
Segurança da Informação
Padrões como Proxy e Decorator permitem adicionar camadas de autenticação, autorização e validação de forma centralizada e controlada, sem expor lógica sensível.
As Três Categorias do GoF
Os 23 padrões catalogados pelo Gang of Four são organizados em três grupos, cada um respondendo a uma pergunta diferente sobre o design do sistema:
Criacionais — Como os objetos são criados?
Controlam e abstraem o processo de criação de objetos, tornando o sistema independente de como seus componentes são instanciados. Exemplos: Singleton, Factory Method, Abstract Factory, Builder, Prototype.
Estruturais — Como os objetos se organizam?
Definem como classes e objetos são compostos para formar estruturas maiores, facilitando a colaboração entre partes independentes do sistema. Exemplos: Adapter, Decorator, Facade, Proxy, Composite, Bridge, Flyweight.
Comportamentais — Como os objetos se comunicam?
Definem como os objetos interagem e distribuem responsabilidades entre si, promovendo baixo acoplamento e alta coesão. Exemplos: Observer, Strategy, Command, Iterator, Template Method, State, Chain of Responsibility.
Criacional — Padrão 01
Singleton
O padrão Singleton garante que uma classe tenha apenas uma única instância em todo o sistema, fornecendo um ponto de acesso global a ela. É amplamente utilizado para gerenciar conexões com banco de dados, configurações de ambiente e sessões de usuário — situações onde ter múltiplas instâncias causaria inconsistência.
Conexão com Integridade: sem o Singleton, cada parte do sistema poderia criar sua própria conexão com o banco de dados, causando múltiplas conexões simultâneas, dados inconsistentes e desperdício de recursos. O Singleton garante que todos compartilhem a mesma instância.
Sem o padrão — o problema: 
// Sem Singleton: cada chamada cria uma nova conexão
class ConexaoBanco {
  constructor() {
    console.log("Nova conexao criada!"); // executado toda vez
    this.conexao = "conectado";
  }
}

const db1 = new ConexaoBanco(); // "Nova conexao criada!"
const db2 = new ConexaoBanco(); // "Nova conexao criada!" — duplicado!

console.log(db1 === db2); // false — sao objetos diferentes
// Problema: dois modulos do sistema operam com conexoes distintas.
// Uma transacao pode ser iniciada em db1 e desfeita em db2.
Com o padrão Singleton — a solução: 
// Com Singleton: apenas uma instancia e criada e compartilhada
class ConexaoBanco {
  constructor() {
    if (ConexaoBanco._instancia) {
      return ConexaoBanco._instancia; // retorna a ja existente
    }
    console.log("Conexao criada pela primeira vez.");
    this.conexao = "conectado";
    ConexaoBanco._instancia = this; // guarda a instancia unica
  }
}

const db1 = new ConexaoBanco(); // "Conexao criada pela primeira vez."
const db2 = new ConexaoBanco(); // sem log — retorna a mesma

console.log(db1 === db2); // true — sao o mesmo objeto
// Garantia: todo o sistema opera sobre a mesma conexao.
Resultado: independente de quantas partes do sistema chamem new ConexaoBanco(), sempre será retornada a mesma instância. A integridade dos dados é preservada.
Criacional — Padrão 02
Factory Method
O padrão Factory Method define uma interface para criação de objetos, mas deixa que as subclasses (ou uma função centralizada) decidam qual tipo de objeto instanciar. O código que usa os objetos não precisa saber como eles foram criados — apenas os utiliza.
Conexão com Qualidade e Usabilidade: o Factory centraliza a lógica de criação. Se as regras de como criar um objeto mudarem, basta alterar a fábrica — não todos os lugares do código que instanciam o objeto. Isso reduz bugs e facilita manutenção.
Sem o padrão — o problema: 
// Sem Factory: logica de criacao espalhada pelo codigo
function processarPagamento(tipo, valor) {
  if (tipo === "cartao") {
    // cria objeto manualmente aqui
    const p = { tipo: "cartao", taxa: 0.03, valor };
    console.log(`Cartao: R$${valor + valor * p.taxa}`);
  } else if (tipo === "pix") {
    // cria objeto manualmente de novo
    const p = { tipo: "pix", taxa: 0, valor };
    console.log(`Pix: R$${valor}`);
  }
  // Se adicionar "boleto", precisa alterar ESTA funcao e
  // todas as outras que criam pagamentos no sistema.
}
Com o padrão Factory Method — a solução: 
// Com Factory: logica de criacao centralizada
class PagamentoCartao {
  calcular(valor) { return valor * 1.03; }
  descricao()    { return "Cartao de Credito"; }
}
class PagamentoPix {
  calcular(valor) { return valor; }
  descricao()    { return "Pix"; }
}
class PagamentoBoleto {
  calcular(valor) { return valor + 2.50; }
  descricao()    { return "Boleto"; }
}

// A fabrica — unico lugar que conhece a logica de criacao
function criarPagamento(tipo) {
  const opcoes = {
    cartao:  () => new PagamentoCartao(),
    pix:     () => new PagamentoPix(),
    boleto:  () => new PagamentoBoleto(),
  };
  if (!opcoes[tipo]) throw new Error(`Tipo invalido: ${tipo}`);
  return opcoes[tipo]();
}

// O codigo que usa nao sabe como o objeto foi criado
const pagamento = criarPagamento("pix");
console.log(pagamento.descricao());      // "Pix"
console.log(pagamento.calcular(100));    // 100

// Para adicionar "criptomoeda", basta criar a classe
// e registra-la na fabrica. Nada mais muda.
Resultado: adicionar um novo tipo de pagamento exige criar uma nova classe e uma linha na fábrica. Zero alteração no código existente. Isso é o Princípio Aberto/Fechado — aberto para extensão, fechado para modificação.
Comportamental — Padrão 03
Observer
O padrão Observer estabelece uma relação de um-para-muitos entre objetos: quando um objeto (o sujeito) muda de estado, todos os seus dependentes (os observadores) são notificados automaticamente. É o fundamento de sistemas de eventos, notificações em tempo real e arquiteturas reativas.
Conexão com Usabilidade: o Observer desacopla quem gera eventos de quem reage a eles. Em sistemas de interface, é o que permite que vários componentes reajam a uma mesma ação do usuário sem que o componente que gerou o evento precise conhecer os demais.
Sem o padrão — o problema: 
// Sem Observer: o Pedido conhece e chama cada modulo manualmente
class Pedido {
  confirmar() {
    console.log("Pedido confirmado!");

    // acoplado diretamente a cada dependente
    EmailService.enviar("Seu pedido foi confirmado");
    Estoque.reservar(this.itens);
    Financeiro.registrar(this.valor);
    // Para adicionar SMS, WhatsApp, Log — alterar ESTA classe
  }
}
Com o padrão Observer — a solução: 
// Com Observer: o Pedido nao conhece quem reage — apenas notifica
class Pedido {
  constructor() {
    this.observadores = [];
  }
  assinar(observador)   { this.observadores.push(observador); }
  notificar(evento)    { this.observadores.forEach(o => o.update(evento)); }

  confirmar() {
    console.log("Pedido confirmado!");
    this.notificar({ tipo: "PEDIDO_CONFIRMADO", dados: this });
  }
}

// Cada observador e independente
const EmailService  = { update: e => console.log("Email enviado!") };
const Estoque       = { update: e => console.log("Estoque reservado!") };
const Financeiro    = { update: e => console.log("Lancamento financeiro!") };
const LogAuditoria  = { update: e => console.log("Log registrado!") };

const pedido = new Pedido();
pedido.assinar(EmailService);
pedido.assinar(Estoque);
pedido.assinar(Financeiro);
pedido.assinar(LogAuditoria);  // basta assinar — sem alterar Pedido

pedido.confirmar();
// "Pedido confirmado!"
// "Email enviado!" / "Estoque reservado!" / "Lancamento financeiro!" / "Log registrado!"
Resultado: a classe Pedido não sabe quem a observa. Para adicionar SMS ou integração com ERP, basta criar um novo observador e assiná-lo — nenhuma linha da classe Pedido é tocada.
Comportamental — Padrão 04
Strategy
O padrão Strategy define uma família de algoritmos, encapsula cada um deles e os torna intercambiáveis. O cliente pode escolher qual algoritmo usar em tempo de execução, sem precisar alterar o código que o utiliza.
Conexão com Qualidade e Usabilidade: evita longas cadeias de if/else para selecionar comportamentos. O código fica mais legível, testável e extensível — cada estratégia é uma unidade isolada e substituível.
Sem o padrão — o problema: 
// Sem Strategy: algoritmos misturados na mesma funcao
function calcularFrete(tipo, peso) {
  if (tipo === "correios") {
    return peso * 5;
  } else if (tipo === "transportadora") {
    return peso * 3 + 10;
  } else if (tipo === "motoboy") {
    return 15;
  }
  // Cada nova transportadora = alterar esta funcao
}
Com o padrão Strategy — a solução: 
// Com Strategy: cada algoritmo e uma funcao/classe independente
const estrategias = {
  correios:       peso => peso * 5,
  transportadora: peso => peso * 3 + 10,
  motoboy:        peso => 15,
  drone:          peso => 8 + peso * 0.5,  // novo: zero impacto no restante
};

function calcularFrete(tipo, peso) {
  const estrategia = estrategias[tipo];
  if (!estrategia) throw new Error(`Tipo desconhecido: ${tipo}`);
  return estrategia(peso);
}

console.log(calcularFrete("correios", 2));       // 10
console.log(calcularFrete("transportadora", 2));  // 16
console.log(calcularFrete("drone", 2));          // 9
Resultado: cada algoritmo de cálculo é isolado e testável individualmente. A função calcularFrete não precisa ser tocada ao adicionar novas transportadoras — basta adicionar uma entrada no objeto de estratégias.
Estrutural — Padrão 05
Proxy
O padrão Proxy fornece um substituto (intermediário) para outro objeto, controlando o acesso a ele. O proxy pode realizar ações antes ou depois de delegar a requisição ao objeto real: verificar permissões, registrar logs, implementar cache ou validar entradas.
Conexão direta com Segurança da Informação: o Proxy é o padrão mais utilizado para implementar controle de acesso, autenticação e autorização de forma centralizada. Em vez de verificar permissões dentro de cada método, o proxy atua como portão de entrada — nada passa sem ser inspecionado.
Sem o padrão — o problema: 
// Sem Proxy: verificacao de permissao duplicada em cada metodo
class BancoDados {
  buscar(usuario, query) {
    // verifica permissao aqui...
    if (!usuario.autenticado) throw new Error("Acesso negado");
    return `Resultado de: ${query}`;
  }
  salvar(usuario, dados) {
    // verifica permissao de novo aqui...
    if (!usuario.autenticado) throw new Error("Acesso negado");
    console.log("Dados salvos");
  }
  // E em TODOS os outros metodos... codigo duplicado, propenso a esquecimento.
}
Com o padrão Proxy — a solução: 
// O objeto real: faz apenas o que e seu — sem verificacoes
class BancoDados {
  buscar(query)  { return `Resultado de: ${query}`; }
  salvar(dados)  { console.log("Dados salvos!"); }
  deletar(id)   { console.log(`Registro ${id} removido!`); }
}

// O Proxy: portao de entrada centralizado para seguranca
class BancoDadosProxy {
  constructor(usuario) {
    this.usuario = usuario;
    this.db      = new BancoDados();
  }

  #verificarAcesso(acao) {
    if (!this.usuario.autenticado)
      throw new Error(`Acesso negado: usuario nao autenticado`);
    if (acao === "deletar" && this.usuario.perfil !== "admin")
      throw new Error("Somente admins podem deletar");
    console.log(`Log: ${this.usuario.nome} executou "${acao}"`);
  }

  buscar(query)  { this.#verificarAcesso("buscar");  return this.db.buscar(query); }
  salvar(dados)  { this.#verificarAcesso("salvar");  this.db.salvar(dados); }
  deletar(id)   { this.#verificarAcesso("deletar"); this.db.deletar(id); }
}

// Uso: o cliente interage com o Proxy, nao com o BD diretamente
const usuario  = { nome: "Ana", autenticado: true, perfil: "user" };
const bancoDB  = new BancoDadosProxy(usuario);

console.log(bancoDB.buscar("SELECT * FROM usuarios"));
// "Log: Ana executou buscar"
// "Resultado de: SELECT * FROM usuarios"

bancoDB.deletar(42);
// Lanca erro: "Somente admins podem deletar"
Resultado: toda a lógica de segurança está em um único lugar — o Proxy. O objeto BancoDados é protegido e nunca acessado diretamente. Logs de auditoria e controle de perfis de acesso são garantidos para todas as operações.
Estrutural — Padrão 06
Decorator
O padrão Decorator permite adicionar comportamentos a um objeto de forma dinâmica, envolvendo-o em "camadas". Cada camada adiciona uma responsabilidade sem modificar o objeto original — como embrulhar um presente: cada papel adiciona algo sem destruir o que está dentro.
Conexão com Qualidade e Segurança: o Decorator é amplamente usado para adicionar validação, sanitização de entradas, compressão e criptografia sobre serviços existentes sem alterar sua implementação original.
Sem o padrão — o problema: 
// Servico basico de envio de mensagem
class Mensagem {
  enviar(texto) { console.log(`Enviando: "${texto}"`); }
}

// Para adicionar criptografia, log e validacao,
// a classe original precisa ser modificada — violando
// o principio de responsabilidade unica.
Com o padrão Decorator — a solução: 
// Objeto base — responsabilidade unica
class Mensagem {
  enviar(texto) { console.log(`Enviando: "${texto}"`); }
}

// Decorator 1: adiciona validacao
function comValidacao(mensagem) {
  return {
    enviar(texto) {
      if (!texto || texto.trim() === "")
        throw new Error("Mensagem nao pode ser vazia");
      mensagem.enviar(texto);
    }
  };
}

// Decorator 2: adiciona criptografia
function comCriptografia(mensagem) {
  return {
    enviar(texto) {
      const textoCript = `[CRIPT]${btoa(texto)}[/CRIPT]`;
      console.log("Texto criptografado.");
      mensagem.enviar(textoCript);
    }
  };
}

// Decorator 3: adiciona log de auditoria
function comLog(mensagem) {
  return {
    enviar(texto) {
      console.log(`[LOG] Envio iniciado as ${new Date().toISOString()}`);
      mensagem.enviar(texto);
      console.log("[LOG] Envio concluido.");
    }
  };
}

// Composicao: empilha as camadas na ordem desejada
const mensagemSegura = comLog(comCriptografia(comValidacao(new Mensagem())));

mensagemSegura.enviar("Dados confidenciais");
// [LOG] Envio iniciado as 2025-06-10T12:00:00.000Z
// Texto criptografado.
// Enviando: "[CRIPT]RGFkb3MgY29uZmlkZW5jaWFpcw==[/CRIPT]"
// [LOG] Envio concluido.
Resultado: a classe Mensagem nunca foi alterada. As camadas de validação, criptografia e log foram adicionadas de fora para dentro. Podem ser removidas, reordenadas ou combinadas de forma diferente a qualquer momento.
Resumo Comparativo dos Padroes
Padrão Categoria Problema que resolve Atributo de qualidade
Singleton Criacional Múltiplas instâncias causando inconsistência de estado Integridade
Factory Method Criacional Lógica de criação de objetos espalhada e acoplada Qualidade / Manutenibilidade
Observer Comportamental Acoplamento direto entre quem emite e quem reage a eventos Usabilidade / Extensibilidade
Strategy Comportamental Algoritmos misturados em longas cadeias de if/else Qualidade / Usabilidade
Proxy Estrutural Verificações de segurança duplicadas em cada método Segurança da Informação
Decorator Estrutural Adicionar funcionalidades exige modificar classes existentes Qualidade / Segurança
Boas práticas ao aplicar Design Patterns
1
Não aplique padrões por aplicar. Design Patterns resolvem problemas reais. Usar um padrão onde não há problema é over-engineering — adiciona complexidade sem benefício. Primeiro identifique o problema, depois avalie se algum padrão o resolve adequadamente.
2
Padrões e testes andam juntos. Código que segue Design Patterns é naturalmente mais testável: cada componente tem uma responsabilidade clara e pode ser testado em isolamento. Strategies, por exemplo, são perfeitas para testes unitários — cada uma é uma função pura e independente.
3
Use os nomes dos padrões como linguagem comum. Quando uma equipe conhece os padrões, a comunicação melhora: em vez de explicar toda a lógica, basta dizer "é um Proxy de autenticação" ou "usamos Observer aqui". Isso reduz mal-entendidos e acelera o desenvolvimento.
4
Combine padrões quando necessário. Padrões não são exclusivos. Um sistema real pode usar Singleton para a conexão com o banco, Factory para criar os modelos, Proxy para controle de acesso e Observer para disparar notificações — tudo junto, cada um no seu lugar.
5
Frameworks modernos já usam padrões. React usa Observer (estado e re-renderização), Express usa Middleware como Decorator, Node.js usa Event Emitter como Observer. Reconhecer esses padrões nas ferramentas que já usa é o próximo passo natural no aprendizado.
