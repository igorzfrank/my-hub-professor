Node.js — JavaScript fora do navegador
Nesta aula você aprenderá o que é o Node.js, como ele funciona internamente, por que ele mudou o desenvolvimento backend, e como criar seus primeiros programas — de scripts simples a um servidor HTTP completo — usando apenas JavaScript.
Capacidades Trabalhadas
 
Aplicar metodologia de desenvolvimento de acordo com o escopo do projeto.
 
Integrar sistemas multiplataformas por meio da linguagem de programação.
O problema que o Node.js resolve
Antes do Node.js, JavaScript era uma linguagem presa dentro do navegador. Se você quisesse construir o backend de um sistema — a parte que processa dados, acessa banco de dados e serve arquivos — precisava aprender outra linguagem: PHP, Java, Python, Ruby...
Além disso, os servidores da época funcionavam com um modelo antigo: para cada conexão de usuário, criavam um novo processo ou thread no sistema operacional. Com milhares de usuários simultâneos, isso consumia memória e CPU de forma insustentável.
Analogia do mundo real: imagine um restaurante onde cada cliente que entra exige um garçom exclusivo que fica parado ao lado da mesa esperando o pedido ser preparado. Com 10 clientes funciona. Com 10.000, você precisa de 10.000 garçons. O Node.js inventou um garçom diferente: ele anota o pedido, leva para a cozinha e vai atender outros clientes enquanto espera — sem ficar parado.
O que é o Node.js?   
O Node.js é um ambiente de execução JavaScript fora do navegador. Criado em 2009 por Ryan Dahl, ele permite rodar código JavaScript diretamente no sistema operacional — no terminal, em servidores, em scripts de automação. Não é uma linguagem nova, não é um framework: é uma plataforma que executa JavaScript.
JavaScript no Navegador
✓ Manipula HTML e CSS
✓ Reage a eventos do usuário
✗ Não acessa o sistema de arquivos
✗ Não cria servidores
✗ Não acessa banco de dados diretamente
JavaScript com Node.js
✓ Cria e lê arquivos no sistema
✓ Cria servidores HTTP
✓ Conecta a bancos de dados
✓ Executa scripts de automação
✓ Consome e serve APIs
Por baixo dos panos — a Engine V8
O Node.js é construído sobre a engine V8 do Google Chrome — o mesmo motor que executa JavaScript no navegador. Ryan Dahl extraiu o V8 do Chrome e o embutiu numa plataforma independente, adicionando APIs para acessar o sistema operacional (arquivos, rede, processos). O resultado é uma ferramenta capaz de executar JavaScript com altíssima performance diretamente no servidor.
Como o Node.js funciona — O Event Loop
O segredo do Node.js é o modelo não-bloqueante (non-blocking) baseado no Event Loop. Enquanto servidores tradicionais bloqueiam uma thread esperando uma resposta do banco de dados, o Node.js registra um callback e continua atendendo outras requisições. Quando a resposta chega, o Event Loop aciona o callback.
Fluxo do Event Loop no Node.js
1
Chamada recebida
Uma requisição HTTP chega. O Node.js registra a operação e a coloca na fila sem bloquear.
 
2
Operação delegada (I/O)
Leitura de arquivo, consulta ao banco ou chamada à rede é delegada ao sistema operacional. O Node.js não espera — segue para a próxima tarefa.
 
3
Event Loop em ação
O Event Loop monitora continuamente a fila de eventos. Enquanto espera, atende outras requisições chegando.
 
4
Callback executado
Quando o sistema operacional sinaliza que a operação terminou, o Event Loop aciona o callback e a resposta é enviada ao cliente.
Servidor tradicional (bloqueante)
1 usuário = 1 thread. 1000 usuários = 1000 threads. Cada thread fica bloqueada esperando respostas do banco. Alto consumo de memória e CPU.
Node.js (não-bloqueante)
1 thread, Event Loop. 1000 usuários = 1000 callbacks registrados. Nenhuma thread fica parada. Baixo consumo de recursos, alta escalabilidade.
Prática 01
Instalação e primeiros passos
O Node.js é instalado pelo site oficial nodejs.org. Sempre baixe a versão LTS (Long Term Support) — ela é a mais estável e recomendada para projetos reais. Após a instalação, o comando node e o gerenciador de pacotes npm ficam disponíveis no terminal.
terminal — verificando a instalação 
# Verifica a versão do Node.js instalada
node --version
# Saída esperada: v20.x.x

# Verifica a versão do NPM (vem junto com o Node)
npm --version
# Saída esperada: 10.x.x

# Abre o REPL interativo do Node (para testar código rapidamente)
node
# Digite qualquer expressão JavaScript e pressione Enter
> 2 + 2
4
# Para sair do REPL: Ctrl+C duas vezes ou .exit
Seu primeiro script Node.js
Crie um arquivo ola.js com o código abaixo e execute com node ola.js no terminal. O Node executa o arquivo de cima para baixo, linha a linha.
ola.js — primeiro script 
// process é um objeto global do Node.js com informações sobre o processo
console.log('Olá, Node.js!');
console.log(`Versão do Node: ${process.version}`);
console.log(`Sistema operacional: ${process.platform}`);
console.log(`Diretório atual: ${process.cwd()}`);

// JavaScript normal — funciona exatamente como no navegador
const numeros = [1, 2, 3, 4, 5];
const soma = numeros.reduce((acc, n) => acc + n, 0);
console.log(`Soma de [1..5]: ${soma}`); // 15
Todo JavaScript válido que você escreveria no navegador funciona no Node.js — exceto o que depende do DOM (document, window, alert). O Node não tem tela — ele tem terminal e sistema de arquivos.
Prática 02
Módulos nativos — o que vem com o Node.js
O Node.js vem com uma biblioteca padrão chamada Core Modules — módulos nativos que você pode usar sem instalar nada. Para utilizá-los, basta importá-los com require() (sistema CommonJS) ou import (ES Modules).
fs
File System
Ler, escrever, copiar e apagar arquivos e pastas no sistema operacional.
path
Caminhos
Montar caminhos de arquivo de forma compatível entre Windows, Mac e Linux.
http
Servidor HTTP
Criar servidores web sem instalar nenhuma biblioteca externa.
os
Sistema Operacional
Informações sobre CPU, memória, plataforma e usuário do sistema.
events
Eventos
Criar emissores e ouvintes de eventos — a base do padrão Observer no Node.
crypto
Criptografia
Hash de senhas, geração de tokens e operações criptográficas sem dependências externas.
fs Trabalhando com arquivos
arquivos.js — leitura e escrita com fs/promises 
// fs/promises: versão do módulo fs que usa Promises (async/await)
const fs = require('fs/promises');

async function gerenciarArquivos() {

  // ── ESCREVER um arquivo ──────────────────────────────────
  const conteudo = 'Olá! Este arquivo foi criado pelo Node.js.\nLinha 2.';
  await fs.writeFile('meu-arquivo.txt', conteudo, 'utf8');
  console.log('Arquivo criado com sucesso!');

  // ── LER um arquivo ───────────────────────────────────────
  const texto = await fs.readFile('meu-arquivo.txt', 'utf8');
  console.log('Conteúdo lido:');
  console.log(texto);

  // ── VERIFICAR se arquivo existe ──────────────────────────
  try {
    await fs.access('outro-arquivo.txt');
    console.log('Arquivo existe!');
  } catch {
    console.log('Arquivo não encontrado.');
  }

  // ── DELETAR um arquivo ───────────────────────────────────
  await fs.unlink('meu-arquivo.txt');
  console.log('Arquivo removido.');
}

gerenciarArquivos();
path Montando caminhos de arquivo
caminhos.js — usando o módulo path 
const path = require('path');

// path.join: monta o caminho correto independente do sistema operacional
// Windows usa \ e Linux/Mac usam / — path.join resolve isso automaticamente
const caminhoArquivo = path.join(__dirname, 'dados', 'usuarios.json');
console.log(caminhoArquivo);
// Saída Linux/Mac: /home/usuario/projeto/dados/usuarios.json
// Saída Windows:  C:\Users\usuario\projeto\dados\usuarios.json

// path.basename: retorna apenas o nome do arquivo
console.log(path.basename(caminhoArquivo)); // 'usuarios.json'

// path.extname: retorna a extensão do arquivo
console.log(path.extname(caminhoArquivo)); // '.json'

// __dirname: variável especial do Node — pasta do arquivo atual
// __filename: caminho completo do arquivo atual
console.log(__dirname);
console.log(__filename);
Use sempre path.join() para montar caminhos em vez de concatenar strings com / ou \. Código que funciona no Linux muitas vezes quebra no Windows por causa do separador de diretórios — path.join() resolve isso automaticamente.
Sistema de Módulos — Organizando o código em arquivos
Em projetos reais, colocar tudo num único arquivo é inviável. O Node.js tem um sistema de módulos que permite dividir o código em arquivos com responsabilidades específicas e importar o que for necessário. Existem dois sistemas: o CommonJS (padrão histórico do Node) e o ES Modules (padrão moderno do JavaScript).
CommonJS — require / module.exports
matematica.js — exportando 
function somar(a, b) {
  return a + b;
}

function multiplicar(a, b) {
  return a * b;
}

// Exporta as funções para uso externo
module.exports = { somar, multiplicar };
app.js — importando 
const { somar, multiplicar }
  = require('./matematica');

console.log(somar(3, 4));       // 7
console.log(multiplicar(3, 4)); // 12
ES Modules — import / export
matematica.mjs — exportando 
export function somar(a, b) {
  return a + b;
}

export function multiplicar(a, b) {
  return a * b;
}

// Ou: export default para exportação principal
app.mjs — importando 
import { somar, multiplicar }
  from './matematica.mjs';

console.log(somar(3, 4));       // 7
console.log(multiplicar(3, 4)); // 12
Qual usar? Em projetos novos com Express.js, o CommonJS (require) ainda é o padrão mais encontrado na documentação e na maioria dos tutoriais. Os ES Modules (import) são o futuro e funcionam bem no Node 18+. O importante é manter o projeto consistente — não misture os dois sistemas no mesmo projeto.
Prática 03
Criando um servidor HTTP com Node puro
O módulo http do Node permite criar um servidor web sem instalar nenhuma dependência externa. Isso mostra a fundação sobre a qual frameworks como o Express.js são construídos — o Express é, essencialmente, uma camada elegante em cima do módulo http.
servidor.js — servidor HTTP com Node puro 
const http = require('http');

// createServer recebe uma função chamada a cada requisição recebida
// req: objeto com informações da requisição (URL, método, headers...)
// res: objeto com métodos para montar e enviar a resposta
const servidor = http.createServer((req, res) => {

  // Loga cada requisição que chega no console do servidor
  console.log(`${req.method} ${req.url}`);

  // Roteamento manual — verifica URL e método da requisição
  if (req.url === '/' && req.method === 'GET') {

    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify({
      mensagem: 'Bem-vindo à API Node.js!',
      versao:   process.version,
    }));

  } else if (req.url === '/usuarios' && req.method === 'GET') {

    const usuarios = [
      { id: 1, nome: 'Ana' },
      { id: 2, nome: 'Bruno' },
    ];
    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify(usuarios));

  } else {

    // Rota não encontrada → 404
    res.writeHead(404, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify({ erro: 'Rota não encontrada' }));
  }
});

// Inicia o servidor na porta 3000
servidor.listen(3000, () => {
  console.log('Servidor rodando em http://localhost:3000');
});
Como testar:
# Execute no terminal:
node servidor.js
# Abra no navegador ou use o Postman:
http://localhost:3000 ← rota principal
http://localhost:3000/usuarios ← lista de usuários
Perceba como o código do servidor com http puro fica verboso à medida que crescem as rotas. É por isso que o Express.js existe: ele abstrai esse roteamento manual e torna o código muito mais limpo — mas por baixo dos panos, é exatamente isso que acontece.
Variáveis de Ambiente — Configuração segura com process.env
Nunca coloque senhas, tokens de API ou configurações sensíveis diretamente no código. Qualquer pessoa com acesso ao repositório verá esses dados. A solução é usar variáveis de ambiente — configurações que ficam fora do código, no sistema operacional ou num arquivo .env que nunca é enviado ao Git.
.env — NUNCA envie este arquivo para o Git 
PORT=3000
DB_HOST=localhost
DB_USUARIO=admin
DB_SENHA=minha_senha_secreta_123
API_KEY=sk-abc123xyz789
config.js — lendo variáveis de ambiente com dotenv 
// Instale o dotenv: npm install dotenv
// Ele carrega as variáveis do arquivo .env para process.env
require('dotenv').config();

// process.env é um objeto global com todas as variáveis de ambiente
const config = {
  porta:    process.env.PORT     || 3000,     // padrão 3000 se não definido
  dbHost:   process.env.DB_HOST  || 'localhost',
  dbSenha:  process.env.DB_SENHA,               // undefined se não definido
  apiKey:   process.env.API_KEY,
};

// Valida se as variáveis obrigatórias estão presentes
if (!config.dbSenha) {
  console.error('ERRO: DB_SENHA não foi definida nas variáveis de ambiente!');
  process.exit(1); // encerra o processo com código de erro
}

module.exports = config;
Segurança obrigatória: adicione .env ao arquivo .gitignore antes do primeiro commit. Senhas e tokens expostos em repositórios públicos são um dos vetores mais comuns de ataques a sistemas em produção. Crie um arquivo .env.exemplo (sem valores reais) para documentar quais variáveis o projeto precisa.
Boas práticas no desenvolvimento com Node.js
1
Nunca coloque credenciais no código. Passwords, tokens de API, chaves secretas e strings de conexão sempre em variáveis de ambiente (.env), nunca diretamente no código.
2
Use sempre async/await com try/catch em operações assíncronas. Erros não tratados em Promises rejeitadas podem derrubar o servidor inteiro.
3
Nunca suba a pasta node_modules para o Git. Use o .gitignore e o package.json para que qualquer pessoa recrie as dependências com npm install.
4
Mantenha o index.js enxuto. O arquivo de entrada do projeto deve apenas inicializar a aplicação — conexão com banco, configurações e inicialização do servidor. A lógica vai nos módulos separados.
5
Use a versão LTS do Node.js em produção. Versões com numeração par são LTS — mantidas com correções de segurança por anos. Versões ímpares são experimentais e têm suporte curto.
