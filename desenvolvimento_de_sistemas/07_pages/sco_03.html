Requisições HTTP — Como os sistemas se comunicam
Nesta aula você aprenderá o que é o protocolo HTTP, como funciona o ciclo de requisição e resposta, quais são os métodos e status codes mais usados, e como fazer requisições a APIs externas com JavaScript — integrando sistemas de plataformas diferentes com código real.
Capacidade Trabalhada
 
Integrar sistemas multiplataformas por meio da linguagem de programação.
O problema que o HTTP resolve
Imagine que seu sistema precisa exibir a previsão do tempo. Você não vai construir satélites meteorológicos — vai pedir essa informação para um sistema que já existe, como a API do OpenWeather. Mas como seu código JavaScript conversa com um servidor que está em outro continente, rodando em outra linguagem, num sistema operacional diferente?
É exatamente isso que o HTTP resolve: ele é o idioma comum que qualquer sistema, em qualquer plataforma, consegue falar e entender. É o protocolo que torna possível a internet funcionar — e é através dele que os sistemas modernos se integram.
Analogia do mundo real: quando você vai ao correio enviar uma encomenda para outro país, existe um protocolo internacional de envio — formulários padronizados, endereçamento num formato universal, rastreamento com status específicos. O HTTP faz exatamente isso para dados digitais: define o formato das mensagens, como endereçar pedidos e como informar o resultado — para que dois sistemas jamais vistos se entendam perfeitamente.
O que é HTTP?
O HTTP — HyperText Transfer Protocol — é o protocolo de comunicação que define como mensagens são formatadas e transmitidas entre um cliente e um servidor na web. Foi criado por Tim Berners-Lee em 1989 no CERN e, desde então, tornou-se a base de toda a comunicação na internet.
HTTP
HyperText Transfer Protocol. Comunicação em texto puro, sem criptografia. Usado em redes internas ou ambientes de desenvolvimento.
HTTPS
HTTP Secure. Todo o tráfego é criptografado com TLS. Obrigatório em sistemas em produção que lidam com dados sensíveis.
Características fundamentais do HTTP
 
Sem estado (Stateless): cada requisição é independente. O servidor não guarda memória de interações anteriores. Se você precisa de contexto (como um usuário logado), usa tokens ou cookies.
 
Cliente-Servidor: sempre há um lado que pede (cliente) e um que responde (servidor). Um sistema pode ser cliente de outro servidor ao mesmo tempo que serve outros clientes.
 
Baseado em texto: as mensagens HTTP são legíveis por humanos — você consegue ler um cabeçalho HTTP e entender o que está sendo comunicado.
O Ciclo: Requisição e Resposta
Toda comunicação HTTP segue um ciclo obrigatório: o cliente envia uma requisição (request) e o servidor devolve uma resposta (response). Nunca é o contrário. O servidor jamais envia dados sem que o cliente tenha pedido primeiro.

Anatomia de uma Requisição HTTP
Uma requisição tem três partes: a linha de início (método + URL + versão do protocolo), os cabeçalhos (headers) com metadados, e o corpo (body) com dados opcionais (usado em POST e PUT).
requisição HTTP — texto puro que trafega pela rede 
// ── LINHA DE INÍCIO ──────────────────────────────────
POST /usuarios HTTP/1.1

// ── CABEÇALHOS (headers) ─────────────────────────────
Host:         api.exemplo.com
Content-Type: application/json    // formato do corpo
Authorization: Bearer eyJhbGci...  // token de autenticação
Accept:        application/json    // formato esperado na resposta

// ── CORPO (body) ─────────────────────────────────────
{
  "nome":  "Ana Souza",
  "email": "ana@email.com"
}
Anatomia de uma Resposta HTTP
A resposta segue estrutura similar: linha de status (versão + código + descrição), cabeçalhos descrevendo o conteúdo, e o corpo com os dados retornados.
resposta HTTP — enviada pelo servidor 
// ── LINHA DE STATUS ──────────────────────────────────
HTTP/1.1 201 Created

// ── CABEÇALHOS ───────────────────────────────────────
Content-Type:  application/json
Location:      /usuarios/42      // onde o novo recurso foi criado

// ── CORPO ────────────────────────────────────────────
{
  "id":    42,
  "nome":  "Ana Souza",
  "email": "ana@email.com"
}
Métodos HTTP — O "verbo" da requisição
O método diz ao servidor o que você quer fazer com o recurso indicado na URL. É a intenção da requisição. Cada método tem uma semântica precisa — usá-los corretamente é o que torna uma API REST bem projetada.
GET
Buscar dados
Solicita a leitura de um recurso. Não modifica nada no servidor. Não tem corpo (body). É a operação mais usada — toda vez que você abre uma página ou consulta uma lista, é um GET. Equivale ao R do CRUD (Read).
GET /produtos    |    GET /produtos/15
POST
Criar um novo recurso
Envia dados no corpo da requisição para criar algo novo no servidor. O servidor define o ID do novo recurso criado. Equivale ao C do CRUD (Create).
POST /produtos    com body: { "nome": "Caderno" }
PUT
Substituir um recurso inteiro
Atualiza um recurso existente substituindo todos os seus campos com os dados enviados. Se um campo não for enviado, é sobrescrito como vazio. Equivale ao U do CRUD (Update completo).
PUT /produtos/15   com body: { campos completos }
PATCH
Atualizar parcialmente um recurso
Atualiza apenas os campos enviados no corpo, sem tocar nos demais. É mais seguro que o PUT quando se quer alterar só um campo específico. Equivale ao U do CRUD (Update parcial).
PATCH /produtos/15   com body: { "preco": 29.90 }
DELETE
Remover um recurso
Remove o recurso identificado na URL. Em geral, não tem corpo. Equivale ao D do CRUD (Delete). Pode retornar 200, 204 (sem conteúdo) ou 404 se não encontrado.
DELETE /produtos/15
Relação entre HTTP e CRUD
Operação CRUD Método HTTP Exemplo de URL Status típico
Create (Criar) POST POST /produtos 201 Created
Read (Ler) GET GET /produtos/15 200 OK
Update (Atualizar) PUT / PATCH PATCH /produtos/15 200 OK
Delete (Remover) DELETE DELETE /produtos/15 200 OK / 204
Status Codes — A resposta do servidor em um número
O status code é um número de 3 dígitos que indica o resultado da requisição. Ele é a primeira coisa que seu código deve verificar ao receber uma resposta. Os status codes são agrupados em 5 famílias, cada uma com um significado distinto.
1xx
Informacional
Processamento em andamento
Raramente usados na prática do dia a dia. Indicam que a requisição foi recebida e o processamento continua. O mais conhecido é o 100 Continue.
2xx
Sucesso
A operação foi bem-sucedida
200 OK  — Sucesso genérico. Usado em GET, PUT, PATCH e DELETE com retorno de dados.
201 Created  — Recurso criado com sucesso. Usado após um POST bem-sucedido.
204 No Content  — Sucesso, mas sem corpo na resposta. Comum no DELETE.
3xx
Redirecionamento
O recurso está em outro lugar
301 Moved Permanently  — O recurso foi movido definitivamente para outra URL.
302 Found  — Redirecionamento temporário.
304 Not Modified  — O recurso não mudou; o cliente pode usar o cache.
4xx
Erro do Cliente
O problema está na requisição enviada
400 Bad Request  — Requisição mal formada (ex: JSON inválido ou campo obrigatório faltando).
401 Unauthorized  — Não autenticado. O cliente precisa se identificar (enviar token).
403 Forbidden  — Autenticado, mas sem permissão para aquele recurso.
404 Not Found  — O recurso não existe naquela URL.
409 Conflict  — Conflito com o estado atual (ex: e-mail já cadastrado).
422 Unprocessable  — Dados entendidos, mas que não passaram na validação de negócio.
5xx
Erro do Servidor
O problema está no servidor
500 Internal Server Error  — Erro genérico no servidor. Geralmente indica um bug no código backend.
502 Bad Gateway  — O servidor recebeu uma resposta inválida de outro servidor.
503 Service Unavailable  — O servidor está fora do ar ou sobrecarregado.
Regra de ouro: erros 4xx são culpa do cliente (quem fez a requisição enviou algo errado). Erros 5xx são culpa do servidor (o backend tem um problema). Nunca use 500 para esconder um erro de validação que deveria ser um 400.
Prática 01
Fazendo requisições com fetch()
A Fetch API é a forma nativa do JavaScript moderno para fazer requisições HTTP. Ela está disponível no navegador e no Node.js (a partir da versão 18). Retorna uma Promise — o que significa que a resposta chega de forma assíncrona e precisamos usar async/await para trabalhar com ela.
GET Buscando dados de uma API pública
buscar-usuario.js — GET com fetch 
// async indica que essa função é assíncrona (pode "esperar" respostas)
async function buscarUsuario(id) {
  try {
    // fetch() inicia a requisição e retorna uma Promise
    // await pausa a função até a resposta chegar
    const resposta = await fetch(`https://jsonplaceholder.typicode.com/users/${id}`);

    // SEMPRE verificar o status antes de processar os dados
    if (!resposta.ok) {
      throw new Error(`Erro ${resposta.status}: ${resposta.statusText}`);
    }

    // .json() converte o corpo da resposta (texto) em objeto JavaScript
    // também retorna uma Promise, por isso precisamos de await
    const usuario = await resposta.json();

    console.log('Usuário encontrado:', usuario.name);
    console.log('E-mail:', usuario.email);

    return usuario;

  } catch (erro) {
    // Captura tanto erros de rede quanto erros lançados manualmente
    console.error('Falha na requisição:', erro.message);
  }
}

// Chamando a função
buscarUsuario(1);
POST Enviando dados para criar um recurso
criar-post.js — POST com fetch 
async function criarPost(titulo, conteudo, autorId) {
  try {
    const resposta = await fetch('https://jsonplaceholder.typicode.com/posts', {

      method: 'POST', // define o método HTTP

      headers: {
        'Content-Type': 'application/json', // avisa ao servidor o formato do corpo
      },

      // body: corpo da requisição — deve ser string JSON
      body: JSON.stringify({
        title:  titulo,
        body:   conteudo,
        userId: autorId,
      }),
    });

    if (!resposta.ok) {
      throw new Error(`Erro ${resposta.status}`);
    }

    const novoPost = await resposta.json();

    // Status 201 = recurso criado com sucesso
    console.log(`Post criado com ID: ${novoPost.id}`);
    return novoPost;

  } catch (erro) {
    console.error('Erro ao criar post:', erro.message);
  }
}

criarPost('Aprendendo HTTP', 'HTTP é o protocolo da web...', 1);
DELETE Removendo um recurso
remover-post.js — DELETE com fetch 
async function removerPost(id) {
  try {
    const resposta = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`, {
      method: 'DELETE',
    });

    // DELETE bem-sucedido geralmente retorna 200 ou 204 (sem conteúdo)
    if (resposta.ok) {
      console.log(`Post ${id} removido. Status: ${resposta.status}`);
    } else {
      console.error(`Não foi possível remover. Status: ${resposta.status}`);
    }

  } catch (erro) {
    console.error('Erro de rede:', erro.message);
  }
}

removerPost(5);
Importante: resposta.ok é true quando o status code está entre 200 e 299. O fetch() não lança erro automaticamente para status 4xx e 5xx — ele só falha se houver um erro de rede. Por isso sempre verifique resposta.ok antes de processar o resultado.
Prática 02
Exemplo completo — Consumindo uma API real
Vamos construir um módulo que consome a API pública JSONPlaceholder (jsonplaceholder.typicode.com) — uma API de testes que simula um sistema real com usuários, posts e comentários. Não precisa de cadastro nem token.
api-posts.js — módulo completo de consumo de API 
// ── CONFIGURAÇÃO ────────────────────────────────────────────
const BASE_URL = 'https://jsonplaceholder.typicode.com';

// Função auxiliar: faz o fetch e trata erros de forma padronizada
async function requisicao(endpoint, opcoes = {}) {
  const resposta = await fetch(`${BASE_URL}${endpoint}`, opcoes);
  if (!resposta.ok) {
    throw new Error(`[${resposta.status}] ${resposta.statusText} — ${endpoint}`);
  }
  // 204 No Content não tem corpo — evita erro ao chamar .json()
  if (resposta.status === 204) return null;
  return resposta.json();
}

// ── OPERAÇÕES CRUD ──────────────────────────────────────────

// READ — busca todos os posts
async function listarPosts() {
  return requisicao('/posts');
}

// READ — busca um post específico pelo ID
async function buscarPost(id) {
  return requisicao(`/posts/${id}`);
}

// CREATE — cria um novo post
async function criarPost(dados) {
  return requisicao('/posts', {
    method:  'POST',
    headers: { 'Content-Type': 'application/json' },
    body:    JSON.stringify(dados),
  });
}

// UPDATE — atualiza um campo do post (PATCH)
async function atualizarPost(id, dados) {
  return requisicao(`/posts/${id}`, {
    method:  'PATCH',
    headers: { 'Content-Type': 'application/json' },
    body:    JSON.stringify(dados),
  });
}

// DELETE — remove um post
async function removerPost(id) {
  return requisicao(`/posts/${id}`, { method: 'DELETE' });
}

// ── EXECUÇÃO E DEMONSTRAÇÃO ─────────────────────────────────
async function demonstracao() {
  console.log('=== 1. Listando os 3 primeiros posts ===');
  const posts = await listarPosts();
  posts.slice(0, 3).forEach(p => console.log(`  [${p.id}] ${p.title}`));

  console.log('\n=== 2. Criando um novo post ===');
  const novo = await criarPost({ title: 'Aprendendo HTTP', body: '...', userId: 1 });
  console.log(`  Post criado com ID: ${novo.id}`);

  console.log('\n=== 3. Atualizando o título do post 1 ===');
  const atualizado = await atualizarPost(1, { title: 'Título atualizado' });
  console.log(`  Novo título: ${atualizado.title}`);

  console.log('\n=== 4. Removendo o post 1 ===');
  await removerPost(1);
  console.log('  Post removido com sucesso.');
}

demonstracao();
Para rodar este exemplo: crie o arquivo api-posts.js, cole o código e execute node api-posts.js no terminal. O Node.js 18+ já suporta fetch() nativamente, sem precisar instalar nada.
Headers HTTP — Os metadados da requisição
Os headers (cabeçalhos) carregam informações sobre a requisição ou a resposta — como o formato dos dados, quem está fazendo o pedido e como o conteúdo deve ser interpretado. São pares de chave-valor enviados antes do corpo da mensagem.
Header Usado em Significado
Content-Type Req. e Resp. Formato do corpo. application/json para APIs REST.
Authorization Requisição Credenciais do cliente. Geralmente Bearer <token> para autenticação por JWT.
Accept Requisição Formato que o cliente aceita na resposta. Informa ao servidor qual tipo de dado retornar.
Cache-Control Resp. e Req. Controla o comportamento de cache. no-cache força buscar sempre do servidor.
CORS Headers Resposta Access-Control-Allow-Origin — controla quais domínios podem acessar a API (segurança do navegador).
requisição autenticada — com header Authorization 
async function buscarPerfil(token) {
  const resposta = await fetch('https://api.exemplo.com/perfil', {
    method: 'GET',
    headers: {
      'Authorization': `Bearer ${token}`, // token JWT do usuário logado
      'Accept':        'application/json',
    },
  });

  // 401 = não autenticado  |  403 = sem permissão
  if (resposta.status === 401) {
    console.error('Token inválido ou expirado. Faça login novamente.');
    return;
  }

  const perfil = await resposta.json();
  console.log('Perfil:', perfil);
}
O que é uma Promise?
Quando seu código JavaScript faz uma requisição HTTP, ele envia a mensagem para a rede e não pode simplesmente parar e esperar — o navegador ou o Node.js precisa continuar executando outras tarefas enquanto a resposta não chega. É aqui que entra a Promise.
Analogia do mundo real: quando você pede uma pizza por aplicativo, o atendente não para tudo para te esperar na porta. Ele te dá um número de pedido — uma promessa de que sua pizza vai chegar. Enquanto isso, ele atende outros clientes. Quando a pizza fica pronta, ele avisa. Uma Promise em JavaScript funciona exatamente assim: é um objeto que representa o resultado futuro de uma operação assíncrona — que ainda não terminou, mas vai terminar.
Os três estados de uma Promise
Uma Promise sempre está em um desses três estados — e só pode transitar de pending para um dos outros dois, nunca voltando atrás.
⏳
pending
Pendente — A operação ainda está em andamento. A requisição foi enviada, mas a resposta ainda não chegou.
✅
fulfilled
Resolvida — A operação concluiu com sucesso. O valor resultante está disponível para uso.
❌
rejected
Rejeitada — A operação falhou. Um erro foi gerado e precisa ser tratado.
Como uma Promise é criada e consumida
Uma Promise recebe uma função com dois parâmetros: resolve (chamado quando tudo deu certo) e reject (chamado quando algo falhou). Quem usa a Promise encadeia .then() para o sucesso e .catch() para o erro.
promise-basica.js — entendendo a estrutura 
// ── CRIANDO uma Promise ──────────────────────────────────────
// Simula uma operação que demora 2 segundos (como uma requisição HTTP)
const minhaPromise = new Promise((resolve, reject) => {

  console.log('Iniciando operação...'); // executa imediatamente

  setTimeout(() => {
    const deuCerto = true; // mude para false para ver o reject

    if (deuCerto) {
      resolve('Dados recebidos com sucesso!'); // → estado: fulfilled
    } else {
      reject(new Error('Falha na operação'));     // → estado: rejected
    }
  }, 2000); // 2 segundos de espera
});

// ── CONSUMINDO com .then() e .catch() ───────────────────────
minhaPromise
  .then((resultado) => {
    // executado quando resolve() foi chamado
    console.log('Sucesso:', resultado);
  })
  .catch((erro) => {
    // executado quando reject() foi chamado
    console.error('Erro:', erro.message);
  })
  .finally(() => {
    // executado SEMPRE, independente do resultado
    console.log('Operação encerrada.');
  });

console.log('Esta linha executa ANTES da Promise terminar!');
// JavaScript não trava — continua executando enquanto espera
async/await — a forma moderna de trabalhar com Promises
Escrever .then().catch() em cadeia funciona, mas pode ficar confuso em operações mais complexas. O async/await é uma sintaxe mais limpa e legível para lidar com Promises — por baixo dos panos, é exatamente a mesma coisa.
sem async/await — difícil de ler 
fetch('/usuarios/1')
  .then(res => res.json())
  .then(usuario => {
    return fetch(`/posts?userId=${usuario.id}`);
  })
  .then(res => res.json())
  .then(posts => {
    console.log(posts);
  })
  .catch(err => console.error(err));
com async/await — limpo e legível 
async function buscarPostsDoUsuario() {
  try {
    const res1    = await fetch('/usuarios/1');
    const usuario = await res1.json();

    const res2  = await fetch(`/posts?userId=${usuario.id}`);
    const posts = await res2.json();

    console.log(posts);
  } catch (err) {
    console.error(err);
  }
}
Regras essenciais do async/await
 
await só pode ser usado dentro de uma função marcada com async. Usar await fora de uma função async é um erro de sintaxe.
 
Uma função async sempre retorna uma Promise, mesmo que você não escreva return new Promise() explicitamente.
 
await "pausa" apenas a função em que está, não o programa inteiro. O restante do código continua executando normalmente.
 
Sempre use try/catch com async/await. É o equivalente ao .catch() das Promises encadeadas — sem ele, erros rejeitados passam silenciosos.
Promise.all — executando múltiplas requisições ao mesmo tempo
Quando você precisa de resultados de várias requisições independentes, fazer uma de cada vez (em sequência) é lento. O Promise.all() dispara todas ao mesmo tempo e espera todas terminarem — o tempo total é o da mais lenta, não a soma de todas.
promise-all.js — requisições paralelas 
async function carregarDashboard() {
  try {
    // As três requisições são disparadas AO MESMO TEMPO
    // Sem Promise.all, cada await esperaria a anterior terminar
    const [usuarios, posts, comentarios] = await Promise.all([
      fetch('https://jsonplaceholder.typicode.com/users')    .then(r => r.json()),
      fetch('https://jsonplaceholder.typicode.com/posts')    .then(r => r.json()),
      fetch('https://jsonplaceholder.typicode.com/comments')  .then(r => r.json()),
    ]);

    // Promise.all aguarda TODAS terminarem — só então continua
    // Se UMA falhar, todas falham (o catch é chamado)
    console.log(`Usuários: ${usuarios.length}`);
    console.log(`Posts: ${posts.length}`);
    console.log(`Comentários: ${comentarios.length}`);

  } catch (erro) {
    console.error('Uma das requisições falhou:', erro.message);
  }
}

carregarDashboard();
Síntese: toda operação assíncrona em JavaScript — incluindo o fetch() — retorna uma Promise. O async/await é apenas uma forma mais legível de consumir Promises. Entender Promises é a base para trabalhar com qualquer operação de rede em JavaScript.
Boas práticas ao trabalhar com HTTP
1
Use o método HTTP correto para cada operação. GET para leitura, POST para criação, PUT/PATCH para atualização e DELETE para remoção. Usar POST para tudo é um anti-padrão que quebra a semântica da API.
2
Sempre verifique o status code antes de processar a resposta. Nunca assuma que uma requisição funcionou. Trate os erros 4xx e 5xx de forma explícita no código.
3
Use try/catch em todas as chamadas assíncronas. Erros de rede (sem conexão, timeout) não geram um status code — eles lançam uma exceção que precisa ser capturada.
4
Nunca exponha informações sensíveis em erros. Não retorne stack traces, queries SQL ou dados internos do sistema em respostas de erro. Retorne mensagens genéricas e registre os detalhes internamente.
5
Sempre use HTTPS em produção. Nunca transmita tokens, senhas ou dados pessoais por HTTP puro. O custo de um certificado SSL é zero (Let's Encrypt) e a exposição de dados sensíveis é inaceitável.
