NPM — Node Package Manager
Nesta aula você aprenderá o que é o NPM, de onde ele veio, por que ele existe e como utilizá-lo no dia a dia do desenvolvimento. Com exemplos práticos, você entenderá como instalar, gerenciar e publicar pacotes — uma habilidade essencial para qualquer desenvolvedor JavaScript moderno.
Capacidades Trabalhadas
 
Aplicar metodologia de desenvolvimento de acordo com o escopo do projeto.
 
Reconhecer e utilizar ferramentas do ecossistema de desenvolvimento de software.
O problema que o NPM resolve
Imagine que você precisa validar e-mails, criptografar senhas, formatar datas e se conectar a um banco de dados no seu projeto. Sem o NPM, você teria duas opções: escrever tudo do zero — o que levaria semanas — ou baixar arquivos de sites aleatórios da internet e colá-los manualmente na sua pasta, sem nenhuma garantia de versão, segurança ou atualizações.
Ambas as opções são lentas, arriscadas e impossíveis de escalar em equipe. Cada desenvolvedor teria uma versão diferente de cada biblioteca, causando bugs impossíveis de reproduzir.
Analogia do mundo real: um arquiteto não fabrica tijolos, canos, fiação elétrica nem janelas. Ele especifica o que precisa e a construtora entrega os materiais certos, na versão certa, no momento certo. O NPM faz exatamente isso para o software: você declara o que precisa e ele entrega — de forma padronizada, versionada e compartilhável com toda a equipe.
O que é o NPM?
O NPM — Node Package Manager — é o gerenciador de pacotes oficial do Node.js. Ele é, ao mesmo tempo, três coisas distintas que trabalham juntas:
01
Um repositório online de pacotes
O site npmjs.com hospeda mais de 2 milhões de pacotes públicos criados por desenvolvedores do mundo inteiro. Qualquer pessoa pode publicar ou baixar código de lá gratuitamente.
02
Uma ferramenta de linha de comando (CLI)
O comando npm que você digita no terminal — ele instala, atualiza, remove pacotes e executa scripts automaticamente no seu projeto.
03
Um sistema de gerenciamento de dependências
Através do arquivo package.json, o NPM registra quais pacotes seu projeto usa e em qual versão — garantindo que todos os membros da equipe trabalhem com o mesmo ambiente.
O que é um pacote?
Um pacote (ou módulo) é um conjunto de arquivos JavaScript que resolve um problema específico e pode ser reutilizado em qualquer projeto. Exemplos: express (servidor web), bcrypt (criptografia de senhas), jest (testes), nodemon (reinicialização automática do servidor).
Origem e contexto histórico
2009
Node.js nasce
Ryan Dahl cria o Node.js, permitindo rodar JavaScript fora do navegador, no servidor.
2010
NPM é lançado
Isaac Z. Schlueter cria o NPM para resolver o problema de compartilhar e reutilizar código Node.js entre projetos.
Hoje
+2 milhões de pacotes
O maior repositório de código open source do mundo, com bilhões de downloads por semana.
O NPM foi criado para resolver um problema real: à medida que o ecossistema Node.js crescia, os desenvolvedores precisavam de uma forma padronizada de compartilhar código. Antes do NPM, cada equipe reinventava a roda — criando soluções para os mesmos problemas repetidamente, sem nenhuma forma de reutilização organizada.
Desde 2020, o NPM pertence à GitHub, que por sua vez pertence à Microsoft. O repositório público continua gratuito. A ferramenta de linha de comando é instalada automaticamente com o Node.js — você já tem o NPM se tem o Node instalado.
Passo a Passo 01
Verificando a instalação
Antes de usar o NPM, confirme que ele está instalado corretamente. Abra o terminal (PowerShell no Windows, Terminal no Mac/Linux) e execute os comandos abaixo. Se os números de versão aparecerem, está tudo pronto.
terminal 
# Verifica a versão do Node.js instalado
node --version
# Saída esperada: v20.x.x (ou superior)

# Verifica a versão do NPM instalado
npm --version
# Saída esperada: 10.x.x (ou superior)
Se o comando não for reconhecido, acesse nodejs.org e baixe o instalador LTS (versão estável). O NPM é instalado automaticamente junto com o Node.js.
Passo a Passo 02
Iniciando um projeto com npm init
Todo projeto Node.js começa com o comando npm init. Ele cria o arquivo package.json — o coração de qualquer projeto NPM. Esse arquivo registra o nome do projeto, a versão, o autor e, mais importante, todas as dependências que o projeto usa.
terminal 
# Cria a pasta do projeto e entra nela
mkdir meu-projeto
cd meu-projeto

# Inicia o NPM com perguntas interativas (nome, versão, etc.)
npm init

# Atalho: responde "sim" para tudo e cria o package.json imediatamente
# Recomendado para iniciantes — você pode editar o arquivo depois
npm init -y
O arquivo package.json gerado pelo npm init -y:
package.json 
{
  "name": "meu-projeto",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",       // arquivo de entrada do projeto
  "scripts": {
    "test": "echo \"Error: no test specified\""
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}
Importante: o package.json deve ser sempre versionado no Git. Ele é o contrato do seu projeto — permite que qualquer pessoa recrie o ambiente com um único comando.
Passo a Passo 03
Instalando pacotes com npm install
O comando npm install (abreviado como npm i) é o mais usado do dia a dia. Quando você instala um pacote, o NPM faz três coisas automaticamente: baixa o código na pasta node_modules, registra o pacote no package.json e cria ou atualiza o package-lock.json.
Dependências de produção — o projeto precisa delas para funcionar
São os pacotes que seu código usa enquanto está em execução. Eles ficam registrados em "dependencies" no package.json.
terminal 
# Instala o Express (framework web) como dependência de produção
npm install express

# Instala múltiplos pacotes de uma vez
npm install express bcrypt dotenv

# Forma abreviada (mesmo resultado)
npm i express bcrypt dotenv
package.json — após a instalação 
{
  "dependencies": {
    "bcrypt": "^5.1.1",   // criptografia de senhas
    "dotenv": "^16.4.5",  // variáveis de ambiente (.env)
    "express": "^4.19.2"  // framework web
  }
}
Dependências de desenvolvimento — usadas só durante o desenvolvimento
São pacotes que ajudam durante o desenvolvimento mas não precisam ir para o servidor de produção. Usam a flag --save-dev e ficam em "devDependencies".
terminal 
# --save-dev indica que é dependência apenas de desenvolvimento
npm install --save-dev jest nodemon

# Forma abreviada
npm i -D jest nodemon
package.json — com devDependencies 
{
  "dependencies": {
    "express": "^4.19.2"
  },
  "devDependencies": {
    "jest":    "^29.7.0",  // framework de testes — só no dev
    "nodemon": "^3.1.0"   // reinicia o servidor automaticamente — só no dev
  }
}
Regra prática: se o servidor de produção precisa do pacote para rodar, é dependencies. Se você usa apenas para programar, testar ou formatar o código, é devDependencies.
Passo a Passo 04
Automatizando tarefas com Scripts NPM
A seção "scripts" do package.json permite criar atalhos para comandos longos e repetitivos. Em vez de digitar o comando completo toda vez, você executa npm run nome-do-script. É uma das funcionalidades mais usadas do NPM no cotidiano profissional.
package.json — seção scripts configurada 
{
  "scripts": {

    // Inicia o servidor normalmente
    "start": "node index.js",

    // Inicia com nodemon: reinicia automaticamente ao salvar arquivos
    "dev": "nodemon index.js",

    // Executa todos os testes com Jest
    "test": "jest",

    // Executa testes e exibe o relatório de cobertura de código
    "test:coverage": "jest --coverage"

  }
}
Como executar cada script:
terminal 
# Inicia o servidor em produção
npm start

# Inicia em modo desenvolvimento (com nodemon)
npm run dev

# Executa os testes
npm test

# Executa os testes com relatório de cobertura
npm run test:coverage
Atenção: npm start e npm test são atalhos especiais do NPM — não precisam da palavra run. Todos os outros scripts precisam de npm run nome.
Passo a Passo 05
Outros comandos essenciais
Além de instalar pacotes, o NPM oferece comandos para gerenciar, atualizar e investigar dependências do projeto. Esses são os mais usados no dia a dia:
terminal — comandos de gerenciamento 
# Instala TODAS as dependências listadas no package.json
# Usado quando você clona um projeto pela primeira vez
npm install

# Remove um pacote do projeto e do package.json
npm uninstall express

# Atualiza um pacote para a versão mais recente compatível
npm update express

# Lista todos os pacotes instalados no projeto
npm list

# Lista apenas os pacotes de primeiro nível (sem sub-dependências)
npm list --depth=0

# Verifica quais pacotes têm versões mais novas disponíveis
npm outdated

# Busca informações sobre um pacote no repositório NPM
npm info express
Referência rápida — os comandos mais usados:
Comando O que faz
npm init -y Cria o package.json com valores padrão
npm install Instala todas as dependências do package.json
npm i <pacote> Instala um pacote como dependência de produção
npm i -D <pacote> Instala um pacote como dependência de desenvolvimento
npm uninstall <pacote> Remove o pacote do projeto e do package.json
npm run <script> Executa um script definido no package.json
npm outdated Mostra quais pacotes têm versões mais novas disponíveis
npm list --depth=0 Lista os pacotes instalados no projeto
Os três arquivos gerados pelo NPM
Quando você usa o NPM, três elementos importantes aparecem na raiz do projeto. Entender o papel de cada um evita confusões muito comuns entre iniciantes:
package.json
O manifesto do projeto. Contém o nome, versão, scripts e a lista de dependências com suas versões permitidas. É o único arquivo que você deve criar e editar manualmente.
Deve ser versionado no Git. É graças a ele que qualquer pessoa pode recriar o projeto.
package-lock.json
O registro exato de todas as versões instaladas — inclusive as sub-dependências (dependências das dependências). Gerado automaticamente pelo NPM, nunca deve ser editado manualmente.
Deve ser versionado no Git. Garante que todos instalem exatamente as mesmas versões.
node_modules/
A pasta onde o NPM baixa fisicamente o código de todos os pacotes instalados. Pode conter centenas de megabytes. É gerada automaticamente ao rodar npm install.
Nunca deve ir para o Git. Adicione node_modules ao .gitignore. Qualquer pessoa consegue recriá-la com npm install.
Exemplo Prático — Projeto completo do zero
Vamos montar um mini projeto Express com variáveis de ambiente e modo de desenvolvimento com nodemon, usando apenas o NPM. Siga os passos em ordem.
Passo 1
Criar e inicializar o projeto
terminal 
mkdir api-npm-exemplo
cd api-npm-exemplo
npm init -y
Passo 2
Instalar as dependências
terminal 
# Dependências de produção: o servidor precisa delas para rodar
npm install express dotenv

# Dependências de desenvolvimento: só usamos na hora de programar
npm install -D nodemon
Passo 3
Configurar os scripts no package.json
package.json 
{
  "name": "api-npm-exemplo",
  "version": "1.0.0",
  "scripts": {
    "start": "node index.js",
    "dev":   "nodemon index.js"
  },
  "dependencies": {
    "dotenv":  "^16.4.5",
    "express": "^4.19.2"
  },
  "devDependencies": {
    "nodemon": "^3.1.0"
  }
}
Passo 4
Criar o arquivo .env e o index.js
.env — variáveis de ambiente (nunca sobe para o Git) 
PORT=3000
AMBIENTE=desenvolvimento
index.js 
// Carrega as variáveis do arquivo .env (pacote dotenv)
require('dotenv').config();

const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.json({
    mensagem: 'API funcionando!',
    ambiente: process.env.AMBIENTE,
    porta:    process.env.PORT,
  });
});

app.listen(process.env.PORT, () => {
  console.log(`Servidor no ar em http://localhost:${process.env.PORT}`);
});
Passo 5
Rodar o projeto
terminal 
# Modo desenvolvimento: reinicia automaticamente ao salvar qualquer arquivo
npm run dev

# Saída esperada:
[nodemon] starting `node index.js`
Servidor no ar em http://localhost:3000
Com nodemon ativo, qualquer alteração salva nos arquivos reinicia o servidor automaticamente. Você não precisa parar e reiniciar manualmente a cada mudança — isso acelera muito o desenvolvimento.
Boas práticas ao usar o NPM
1
Nunca suba o node_modules para o Git. Crie um arquivo .gitignore na raiz do projeto e adicione node_modules nele. Qualquer pessoa recria a pasta com npm install.
2
Separe corretamente as dependências. Ferramentas de desenvolvimento (jest, nodemon, eslint) pertencem às devDependencies. Carregar pacotes desnecessários em produção aumenta o tamanho do servidor e a superfície de ataque.
3
Verifique os pacotes antes de instalar. Antes de instalar um pacote desconhecido, visite npmjs.com e verifique: quando foi atualizado pela última vez, quantos downloads semanais tem e se a documentação é clara.
4
Mantenha as dependências atualizadas. Use npm outdated regularmente para verificar quais pacotes têm versões mais novas com correções de segurança.
5
Use scripts para padronizar o projeto. Todo colaborador deve iniciar o servidor, rodar os testes e construir o build do mesmo jeito. Centralizar isso em scripts NPM evita o problema de "funciona na minha máquina".
