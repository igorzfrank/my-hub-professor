Padrão MVC com Express.js
Nesta aula você aprenderá o que é o padrão arquitetural MVC, por que ele existe e como aplicá-lo em um projeto Express.js real. Cada camada será explicada com exemplos progressivos e, ao final, veremos como o MVC se conecta com os Design Patterns que estudamos anteriormente.
Capacidade Trabalhada
 
Aplicar metodologia de desenvolvimento de acordo com o escopo do projeto.
O problema que o MVC resolve
Imagine que você está construindo uma API para gerenciar tarefas. No começo, é tentador colocar tudo em um arquivo só: os dados, as regras de negócio, o tratamento das requisições e a formatação das respostas. Funciona — por um tempo.
Conforme o projeto cresce, esse arquivo vira um labirinto. Qualquer alteração pode quebrar algo inesperado. Novos desenvolvedores não entendem onde cada coisa está. Isso tem um nome: código espaguete.
sem-mvc.js — tudo no mesmo lugar (o problema) 
// Dados, regras, requisição e resposta — misturados num único arquivo.
// Conforme o sistema cresce, isso se torna impossível de manter.
const express = require('express');
const app = express();
app.use(express.json());

// Dados aqui...
let tarefas = [{ id: 1, titulo: 'Estudar', concluida: false }];

// Regras de negócio aqui...
// Tratamento de requisição aqui...
// Validação aqui...
// Resposta aqui... tudo junto, tudo misturado.
app.get('/tarefas', (req, res) => {
  // E se eu precisar reutilizar essa lista em outra rota?
  // E se outro dev precisar entender o que é "regra" e o que é "rota"?
  res.json(tarefas);
});

app.listen(3000);
Analogia do mundo real: pense numa cozinha onde o mesmo funcionário anota o pedido, cozinha, monta o prato, cobra e lava a louça. Com um cliente funciona. Com cinquenta clientes, vira caos. A solução — tanto na cozinha quanto no software — é separar responsabilidades: cada pessoa tem um papel claro e delimitado.
O que é o padrão MVC?
Model — View — Controller
O MVC é um padrão arquitetural que divide o projeto em três camadas com responsabilidades bem definidas e que não se misturam. Criado nos anos 1970 no Xerox PARC, tornou-se o padrão mais adotado no desenvolvimento web — presente em frameworks como Express.js, Laravel, Django e Ruby on Rails.
M
Model — Modelo
É onde vivem os dados e todas as regras de negócio. O Model é responsável por buscar informações no banco de dados, validar entradas, aplicar cálculos e garantir que os dados estejam corretos antes de qualquer coisa. Ele não sabe que existe uma requisição HTTP — trabalha apenas com dados.
Na cozinha: é o cozinheiro e a receita. Sabe o que pode ser preparado, como deve ser feito e garante que o prato saia correto. Não atende o cliente diretamente.
V
View — Visualização
É a camada de apresentação — o que o usuário ou o sistema consumidor recebe como resposta. Em sistemas web tradicionais, a View é o HTML renderizado na tela. Em APIs REST (como vamos construir com Express.js), a View é o JSON retornado pelo servidor. Ela apenas exibe dados, nunca processa nada.
Na cozinha: é o balcão de entrega. O cliente recebe o prato pronto — sem ver a cozinha, sem saber a receita. Apenas o resultado final, apresentado de forma adequada.
C
Controller — Controlador
É o intermediário que orquestra o fluxo. O Controller recebe a requisição HTTP da rota, decide quais dados pedir ao Model, e monta a resposta que será entregue como View. Ele sabe o que fazer, mas delega ao Model como fazer. Se o Controller está ficando muito grande, é sinal de que lógica de negócio escapou para o lugar errado.
Na cozinha: é o garçom. Recebe o pedido do cliente, leva para a cozinha (Model), busca o resultado e entrega no balcão (View). Não cozinha e não é o balcão — apenas coordena o fluxo.

Estrutura de Pastas do Projeto
O MVC não é apenas um conceito — ele precisa se refletir na organização física dos arquivos. Cada pasta representa uma camada, e essa separação deve ser respeitada rigorosamente. Um arquivo que está na pasta errada é um sinal de que a responsabilidade está no lugar errado. 
projeto-tarefas/
│
├── index.js              ← Ponto de entrada: inicializa o servidor e registra as rotas
│
├── routes/               ← URLs da API — apenas mapeia caminhos para controllers
│   └── tarefasRoutes.js
│
├── controllers/          ← Camada C: recebe req, consulta Model, devolve res
│   └── tarefasController.js
│
└── models/               ← Camada M: dados, validações e regras de negócio
    └── tarefasModel.js
routes/
Define quais URLs existem e qual método HTTP cada uma aceita (GET, POST, PUT, DELETE). Não contém lógica alguma — apenas aponta para o controller correto. Se você encontrar um if dentro de uma rota, algo está no lugar errado.
ctrls/
Recebe o req da rota, chama as funções do Model, trata erros e monta o res. Não armazena dados e não acessa o banco diretamente. Se o controller está crescendo muito, é porque regras de negócio escaparam para ele.
models/
Contém os dados e toda a lógica de negócio: validações, cálculos, consultas ao banco. O Model desconhece o Controller e a View — ele só lida com dados. Isso é o que permite trocar o banco de dados sem alterar nenhuma outra camada.
Exemplo Completo — API de Tarefas
Vamos construir uma API de gerenciamento de tarefas do zero, seguindo o MVC de forma correta. Leia os arquivos na ordem e observe como cada camada tem uma responsabilidade exclusiva — e como elas se comunicam sem se misturar.
Arquivo 1 de 4
index.js — Ponto de Entrada do Servidor
O arquivo principal deve ser o mais enxuto possível. Sua única função é iniciar o servidor Express, registrar os middlewares essenciais e conectar as rotas. Nenhuma lógica de negócio deve aparecer aqui.
index.js 
// 1. Importa o framework
const express = require('express');
const app     = express();

// 2. Importa o arquivo de rotas
const tarefasRoutes = require('./routes/tarefasRoutes');

// 3. Middleware: habilita leitura do corpo da requisição em JSON
//    Sem isso, req.body seria undefined no controller
app.use(express.json());

// 4. Registra as rotas sob o prefixo /tarefas
//    GET /tarefas, POST /tarefas, etc. passarão por aqui
app.use('/tarefas', tarefasRoutes);

// 5. Inicia o servidor
app.listen(3000, () => {
  console.log('Servidor rodando em http://localhost:3000');
});
Arquivo 2 de 4
routes/tarefasRoutes.js — Mapa de URLs
As rotas funcionam como um índice: dizem qual URL faz o quê, mas não executam nada por conta própria. Cada linha associa um método HTTP e um caminho a uma função do controller. Nenhuma linha de lógica aqui.
routes/tarefasRoutes.js 
const express           = require('express');
const router             = express.Router();
const tarefasController  = require('../controllers/tarefasController');

// Cada linha = um "endereço" da API e quem vai atendê-lo
router.get('/',    tarefasController.listar);      // GET  /tarefas
router.get('/:id', tarefasController.buscarPorId); // GET  /tarefas/1
router.post('/',    tarefasController.criar);       // POST /tarefas
router.put('/:id', tarefasController.atualizar);   // PUT  /tarefas/1
router.delete('/:id', tarefasController.remover); // DELETE /tarefas/1

module.exports = router;
Arquivo 3 de 4
models/tarefasModel.js — Dados e Regras de Negócio
Este é o arquivo mais importante do ponto de vista das regras. Observe que ele não conhece o Express — não há req, não há res. Ele apenas lida com os dados, aplica validações e retorna resultados. Isso o torna totalmente testável de forma isolada com Jest.
models/tarefasModel.js 
// Simula um banco de dados em memória
// Em um projeto real, aqui entraria o Prisma, Sequelize ou outra ORM
let tarefas = [
  { id: 1, titulo: 'Estudar MVC',      concluida: false },
  { id: 2, titulo: 'Criar projeto API', concluida: true  },
];
let proximoId = 3;

// Retorna todas as tarefas cadastradas
const listarTodas = () => tarefas;

// Busca uma tarefa específica pelo ID numérico
const buscarPorId = (id) =>
  tarefas.find(t => t.id === Number(id));

// Cria uma nova tarefa — aqui fica a validação, não no controller
const criar = ({ titulo }) => {
  if (!titulo || titulo.trim() === '')
    throw new Error('O título da tarefa é obrigatório.');

  const nova = { id: proximoId++, titulo: titulo.trim(), concluida: false };
  tarefas.push(nova);
  return nova;
};

// Atualiza título e/ou status de uma tarefa existente
const atualizar = (id, dados) => {
  const tarefa = buscarPorId(id);
  if (!tarefa) return null;
  if (dados.titulo    !== undefined) tarefa.titulo    = dados.titulo.trim();
  if (dados.concluida !== undefined) tarefa.concluida = dados.concluida;
  return tarefa;
};

// Remove uma tarefa e retorna true se bem-sucedido, false se não encontrou
const remover = (id) => {
  const antes = tarefas.length;
  tarefas = tarefas.filter(t => t.id !== Number(id));
  return tarefas.length < antes;
};

module.exports = { listarTodas, buscarPorId, criar, atualizar, remover };
Arquivo 4 de 4
controllers/tarefasController.js — O Orquestrador
O Controller é o único arquivo que lida simultaneamente com o req e o res do Express. Ele extrai o que precisa da requisição, delega ao Model, trata erros e devolve a resposta com o código de status HTTP correto.
controllers/tarefasController.js 
const Tarefa = require('../models/tarefasModel');

// GET /tarefas — retorna todas as tarefas
exports.listar = (req, res) => {
  const tarefas = Tarefa.listarTodas();
  res.status(200).json(tarefas);
};

// GET /tarefas/:id — busca uma tarefa pelo ID da URL
exports.buscarPorId = (req, res) => {
  const tarefa = Tarefa.buscarPorId(req.params.id);
  if (!tarefa)
    return res.status(404).json({ erro: 'Tarefa não encontrada.' });
  res.status(200).json(tarefa);
};

// POST /tarefas — cria nova tarefa com os dados do corpo da requisição
exports.criar = (req, res) => {
  try {
    const nova = Tarefa.criar(req.body);
    // 201 = Created: recurso criado com sucesso
    res.status(201).json({ mensagem: 'Tarefa criada!', tarefa: nova });
  } catch (erro) {
    // O Model lançou um erro de validação — respondemos 400
    res.status(400).json({ erro: erro.message });
  }
};

// PUT /tarefas/:id — atualiza os dados de uma tarefa existente
exports.atualizar = (req, res) => {
  const atualizada = Tarefa.atualizar(req.params.id, req.body);
  if (!atualizada)
    return res.status(404).json({ erro: 'Tarefa não encontrada.' });
  res.status(200).json({ mensagem: 'Tarefa atualizada!', tarefa: atualizada });
};

// DELETE /tarefas/:id — remove a tarefa permanentemente
exports.remover = (req, res) => {
  const removida = Tarefa.remover(req.params.id);
  if (!removida)
    return res.status(404).json({ erro: 'Tarefa não encontrada.' });
  res.status(200).json({ mensagem: 'Tarefa removida com sucesso!' });
};
Note os códigos de status HTTP: 200 sucesso, 201 criado, 400 dado inválido, 404 não encontrado. Isso torna a API previsível e profissional para qualquer sistema que a consuma.

MVC e Design Patterns — Qual é a relação?
Conexão com a aula anterior
O MVC e os Design Patterns que estudamos não são concorrentes — eles trabalham em níveis diferentes e se complementam. O MVC organiza o projeto inteiro em camadas. Os Design Patterns resolvem problemas específicos dentro dessas camadas. Na prática, você usará os dois ao mesmo tempo, sem perceber.
Singleton
Garante que a conexão com o banco de dados — usada pelos Models — exista em apenas uma instância no sistema inteiro. Evita múltiplas conexões abertas ao mesmo tempo.
Factory Method
Quando o Model precisa criar diferentes tipos de tarefas (urgente, comum, recorrente), a Factory centraliza essa lógica. O Controller não precisa saber como cada tipo é criado — só recebe o objeto pronto.
Observer
Quando uma tarefa é criada, o Model pode notificar outros módulos (e-mail, log, dashboard) sem conhecê-los. No Express, isso acontece via eventos do Node.js — o Controller dispara, os observadores reagem.
Proxy e Decorator
São exatamente o que o Express chama de middlewares. Uma função autenticar antes do Controller é um Proxy. Uma função de log que envolve a rota é um Decorator. Você já usa Design Patterns sem perceber.
O MVC diz onde cada coisa deve estar. Os Design Patterns dizem como resolver os problemas dentro de cada lugar. Aprender os dois é o que transforma código funcional em código profissional.
Boas práticas ao usar o padrão MVC
1
Model não conhece o Controller, e muito menos a View. Se você precisar importar o req ou res dentro do Model, é um sinal de que a arquitetura está sendo violada.
2
Validações pertencem ao Model, não ao Controller. O Controller só verifica se o Model retornou algo ou lançou um erro — ele não decide se um dado é válido.
3
Controllers pequenos são um bom sinal. Se um controller tem muitas linhas, provavelmente está assumindo responsabilidades do Model. Refatore.
4
Sempre retorne códigos de status HTTP corretos. Eles são a "linguagem" da API. 200, 201, 400, 404 têm significados específicos que sistemas e desenvolvedores esperam.
5
Models isolados são facilmente testáveis. Funções que só recebem dados e retornam dados — sem req ou res — são testadas com Jest de forma simples e direta, como praticamos nas aulas de testes.
