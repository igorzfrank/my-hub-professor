1 🧠 Organização de Projetos ReactJs
📌 Por que organizar?
❌ Sem organização
Código bagunçado → difícil manter
Componentes misturados → confuso
Reutilização baixa → retrabalho
✔ Com organização
Organização → escalabilidade
Clareza → produtividade
Padronização → trabalho em equipe
📁 Estrutura básica recomendada
src/
├── components/
├── pages/
├── hooks/
├── services/
├── assets/
├── App.jsx
└── main.jsx
components/ Componentes reutilizáveis: botões, cards, inputs
pages/ Páginas principais da aplicação
hooks/ Funções customizadas (avançado)
services/ Requisições externas (API)
assets/ Imagens, ícones, arquivos estáticos
2 🧩 Separação de Componentes
📌
Cada componente deve ter uma única responsabilidade.
❌ Errado — tudo em um componente
function App() {
  return (
    <div>
      <h1>Lista</h1>
      <input />
      <button>Adicionar</button>
      <ul><li>Tarefa 1</li></ul>
    </div>
  );
}
✔ Correto — componentes separados
App
├── Header
├── Form
├── Lista
└── Item
♻️ Reutilização
🧪 Testabilidade
🔧 Manutenção fácil
3 🧼 Boas Práticas no React
🧩 1. Nomeação clara
✔ function ListaTarefas() {}
❌ function L() {}
🧩 2. Componentes pequenos
✔ function Botao() {}
❌ Evitar componentes gigantes
🧩 3. Separar lógica e interface
lógica → estado, funções
interface → JSX
🧩 4. Evitar duplicação
Se você escreveu duas vezes, provavelmente deveria ser um componente.
🧩 5. Sempre usar key em listas
{lista.map(item => (
  <Item key={item.id} />
))}
🧩 6. Estado mínimo necessário
Não criar estados desnecessários. Crie estado apenas para dados que realmente mudam e afetam a interface.
4 🔄 Revisão Geral dos Conceitos
Props 📦 Comunicação
• Comunicação entre componentes
• Fluxo: pai → filho
useState ⚛️ Estado
• Controla dados dinâmicos
• Re-renderiza a interface
useEffect ⚙️ Efeitos
• Executa efeitos colaterais
• Simula chamadas de API
• Reage a mudanças de estado
Render + Events 🔁 UI
• Listas com map()
• Condicional com if ou ? :
• Eventos: onClick · onChange
